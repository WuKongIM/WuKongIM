# Web 多语言设计

- 日期：2026-04-23
- 范围：为 `web/` 管理台新增可扩展的前端国际化能力，首批支持中文与英文，并覆盖语言切换、语言偏好持久化、本地格式化和关键页面文案迁移
- 关联目录：
  - `web/src/app/`
  - `web/src/lib/navigation.ts`
  - `web/src/pages/`
  - `web/src/components/`
  - `web/src/test/`
  - `docs/superpowers/specs/2026-04-22-web-admin-shell-design.md`
  - `docs/superpowers/specs/2026-04-22-web-manager-api-integration-design.md`

## 1. 背景

当前 `web/` 管理台已经具备以下能力：

- `AppShell + SidebarNav + Topbar` 的统一壳子
- 登录页、路由保护、JWT 登录态管理
- `dashboard`、`nodes`、`channels`、`connections`、`slots`、`network`、`topology` 等页面
- 基于 Vitest + Testing Library 的前端测试基线

但当前页面文案仍全部写死为英文，且文案分散在以下位置：

- `web/src/pages/*`
- `web/src/app/layout/*`
- `web/src/lib/navigation.ts`
- `web/src/components/*`

这会带来几个问题：

1. 当前无法满足中文使用场景
2. 后续扩展更多语言时，页面内直接写死字符串会导致维护成本快速上升
3. 顶栏、登录页、导航、错误提示、状态提示、时间格式等都缺少统一的 locale 入口

因此本轮需要为 `web/` 引入正式的国际化框架与基础设施，而不是继续在组件里散落 `if/else` 语言判断。

## 2. 目标与非目标

### 2.1 目标

本次设计目标如下：

1. 为 `web/` 引入成熟 i18n 框架，首选 `react-intl`
2. 首批支持 `zh-CN` 和 `en`
3. 默认跟随浏览器语言，但允许用户手动切换
4. 手动切换后的语言偏好持久化到本地存储
5. 在登录页和登录后的 `Topbar` 都提供语言切换入口
6. 页面切换语言时即时生效，不刷新页面、不跳转路由
7. 覆盖静态文案、前端生成文案和本地格式化内容
8. 为关键语言行为与关键页面补齐单元测试
9. 保持扩展性，使后续新增第三种语言时不需要重构现有结构

### 2.2 非目标

当前阶段明确不做以下内容：

- 不引入 URL 级语言路由，例如 `/en/dashboard`
- 不把语言偏好同步到服务端或用户资料
- 不实现运行时异步加载语言包
- 不翻译后端原始响应中的自由文本内容
- 不为 `react-intl` 之外再叠加一层复杂 DSL 或远程词典系统
- 不在本轮支持中文与英文之外的第三种语言，只保留可扩展结构

## 3. 方案选择

本轮考虑了以下三种方案：

### 3.1 方案一：组件内手写中英文判断

做法是在页面或组件内部直接根据当前语言写 `zh/en` 分支。

优点：

- 实现速度最快
- 不需要引入第三方依赖

缺点：

- 文案会持续散落在组件内部
- 导航、页面、错误提示、格式化逻辑难以统一
- 后续扩展第三种语言时修改面过大

结论：不采用。该方案只适合短期补丁，不适合当前“先支持两种语言并后续可扩展”的目标。

### 3.2 方案二：轻量自研 i18n 层

做法是在 `web/src/i18n/` 中自建 locale store、词典和 `t()` 封装，不依赖成熟 i18n 库。

优点：

- 依赖少
- 对当前项目规模足够可用

缺点：

- 消息格式、日期时间、数字格式化能力需要自己补齐
- 随着页面增长，插值、格式化、测试习惯会逐渐变成隐性维护成本

结论：不采用。该方案可行，但用户已经明确选择成熟方案。

### 3.3 方案三：引入成熟 i18n 框架

做法是在 `web/` 中接入 `react-intl`，统一处理文本消息、插值、日期时间格式化和 locale 上下文。

优点：

- 组件接入简洁，和当前 React 架构适配度高
- 文本翻译与本地格式化可使用同一套上下文
- 首批两种语言接入成本可控，后续扩展能力充足
- 测试与文案迁移路径清晰

缺点：

- 会增加一个依赖
- 需要对现有静态导航与页面文案做一次系统化迁移

结论：采用该方案，并使用 `react-intl` 作为本轮国际化基础设施。

## 4. 设计原则

### 4.1 语言能力集中化，页面不自行判断语言

页面和组件不直接根据 `locale` 写条件分支，而是统一通过国际化上下文与消息词典获取用户可见文案。

这样可以保证：

- 新增语言时只需要补消息词典
- 页面逻辑不因语言数量增加而膨胀
- 顶栏、导航、页面、弹窗、错误提示可以统一风格和命名方式

### 4.2 文本翻译和本地格式化使用同一上下文

本轮不只切换静态文字，也要让前端生成的日期时间格式跟随当前语言。因此文本消息、插值消息和时间格式化都由 `react-intl` 的同一 locale 上下文提供。

### 4.3 语言偏好与登录态解耦

语言偏好是界面层设置，不应绑定到登录态生命周期：

- 退出登录时不清理语言选择
- 未登录时也能切换语言
- 重新进入页面时优先沿用用户上次手动选择

### 4.4 可扩展但不过度设计

本轮只做支持中英文所需的基础设施，不预留复杂多租户、多命名空间、远程词典和动态模块加载等超前能力。目录与接口保持清晰，但不做过重抽象。

## 5. 目录与模块设计

建议在 `web/src/` 下新增 `i18n/` 目录，结构如下：

```text
web/src/i18n/
  constants.ts
  detect-locale.ts
  locale-store.ts
  provider.tsx
  use-locale.ts
  types.ts
  messages/
    en.ts
    zh-CN.ts
```

各文件职责如下：

- `types.ts`
  - 定义 `AppLocale`
  - 当前仅支持 `"en" | "zh-CN"`
- `constants.ts`
  - 定义支持语言集合
  - 定义默认语言
  - 定义本地存储 key
- `detect-locale.ts`
  - 从 `localStorage` 与 `navigator.language` 解析初始语言
  - 负责浏览器语言归一化与 fallback 规则
- `messages/en.ts`、`messages/zh-CN.ts`
  - 分语言导出消息词典
  - 仅放用户可见文案，不放内部常量
- `locale-store.ts`
  - 保存当前语言
  - 暴露切换语言方法
  - 持久化用户手动选择
- `provider.tsx`
  - 将 locale store 与 `IntlProvider` 连接
  - 向整个应用提供国际化上下文
- `use-locale.ts`
  - 给语言切换器等 UI 暴露当前语言与切换方法

## 6. 应用接入方式

### 6.1 Provider 接入

当前 `web/src/app/providers.tsx` 负责挂载全局 provider。国际化接入方式如下：

1. 新增 `I18nProvider`
2. 在 `AppProviders` 中将 `TooltipProvider` 放到 `I18nProvider` 内部或外部，但保证整个应用都能读取 locale 上下文
3. `web/src/App.tsx` 保持结构稳定，不额外感知国际化细节

这样国际化能力只在 provider 层装配一次，页面层不感知底层来源。

### 6.2 状态初始化优先级

初始语言按以下优先级确定：

1. 本地存储中的用户手动选择
2. 浏览器语言 `navigator.language`
3. 默认回退到 `en`

### 6.3 浏览器语言归一化

浏览器语言判断采用宽松匹配：

- `zh`、`zh-CN`、`zh-Hans`、`zh-SG` 等统一归到 `zh-CN`
- 其他当前未支持的语言统一回退到 `en`

该策略能在本轮只支持两种语言时保持用户体验稳定，同时便于后续扩展更多 locale。

## 7. 文案模型与命名约定

### 7.1 消息 key 命名

消息 key 按界面语义组织，而不是按文件名自由散落。建议采用以下风格：

- `common.refresh`
- `common.search`
- `common.logout`
- `auth.signIn`
- `auth.username`
- `auth.password`
- `auth.invalidCredentials`
- `nav.group.overview`
- `nav.dashboard.title`
- `nav.dashboard.description`
- `dashboard.generatedAt`
- `dashboard.controllerLeader`

### 7.2 导航元信息迁移

当前 `web/src/lib/navigation.ts` 直接保存英文标题和描述。本轮改为保存 message id，而不是最终显示文本。

建议调整为：

- 导航结构仍然保留 `href`、`icon` 等稳定字段
- `title`、`description` 改为 `titleMessageId`、`descriptionMessageId`
- `group.label` 改为 `labelMessageId`

实际渲染时：

- `SidebarNav` 负责按当前 locale 渲染导航名称
- `Topbar` 通过同一份元信息渲染当前页面标题和描述

这样同一个路由的标题与描述只维护一份消息定义，避免 sidebar 和 topbar 文案分叉。

### 7.3 翻译范围边界

本轮词典只覆盖“用户可见文案”，包括：

- 页面标题、描述、副标题
- 按钮、表头、输入标签、空态和错误提示
- 前端拼装出的状态前缀或标签
- 语言切换器本身的标签

不进入词典的内容包括：

- 路由 path
- API 字段名
- 测试 fixture 的后端原始响应字段
- 内部枚举常量名

## 8. 页面与组件改造范围

### 8.1 必须覆盖的壳层文案

本轮优先覆盖整个应用都会出现的壳层文案：

- `web/src/app/layout/topbar.tsx`
- `web/src/app/layout/sidebar-nav.tsx`
- `web/src/lib/navigation.ts`
- 可能包含全局空态/错误态的公共组件

这样即使某些业务页还未完全迁移，整体管理台的主框架也能正确切换语言。

### 8.2 必须覆盖的页面

首批至少覆盖：

- 登录页 `web/src/pages/login/page.tsx`
- Dashboard `web/src/pages/dashboard/page.tsx`
- 其余已存在页面的标题、说明、空态和关键操作文案

考虑到当前 `web/` 已存在页面数量有限，本轮建议把现有页面中所有用户可见静态文案一并迁移，而不是只迁移一两个页面后留下明显混杂状态。

### 8.3 前端生成文案

这类文案本轮也需要统一国际化，包括但不限于：

- 登录错误提示
- `Refreshing...`
- `Generated:`
- `Controller leader:`
- `Scope: single-node cluster`
- `Generated: pending`

其中“单节点集群”表述需继续遵守仓库运行语义约束，不使用独立单机语义。

## 9. 语言切换交互设计

### 9.1 切换入口位置

语言切换入口放两处：

1. 登录页
2. 登录后的 `Topbar`

这样未登录与已登录场景都能切换语言，且用户无需先登录才能调整界面语言。

### 9.2 切换行为

切换语言后应满足：

- 即时生效
- 不刷新页面
- 不跳转当前路由
- 当前页面局部状态尽量保持不变
- 立即写入 `localStorage`

### 9.3 切换器形式

第一版使用简单、可读的显式切换，例如：

- `中文`
- `EN`

具体视觉可采用轻量按钮组、下拉菜单或 segmented control，但不应增加复杂交互。由于当前管理台视觉偏克制，推荐使用与现有 `Button` 风格兼容的紧凑切换器。

## 10. 本地格式化策略

### 10.1 日期时间

当前页面中存在 `new Date(value).toLocaleString()` 一类格式化逻辑。本轮应改为基于当前 locale 的统一格式化，而不是继续直接依赖浏览器默认环境。

目标效果如下：

- 当前语言为中文时，时间按中文 locale 格式显示
- 当前语言为英文时，时间按英文 locale 格式显示
- 用户手动切换语言后，已有页面中的时间展示同步更新

### 10.2 插值型消息

对于 `Generated: {time}`、`Controller leader: {id}` 这类前缀加动态值的文案，统一使用消息模板与变量插值，不在组件中手写拼接。

### 10.3 状态值映射

接口返回的状态枚举值保持原样，不改 API 层契约；如果页面需要面向用户显示更友好的状态文本，则在前端视图层建立“状态值 -> message id”映射。

## 11. 测试策略

本轮遵循现有 `Vitest + Testing Library` 测试基线，并优先覆盖核心语言行为，而不是对所有页面做大面积快照。

### 11.1 核心行为测试

至少新增或调整以下测试：

1. locale 检测
   - 有本地偏好时优先使用本地偏好
   - 无本地偏好时按浏览器语言选择
   - 未知浏览器语言回退到 `en`
2. 语言切换持久化
   - 登录页切换语言后写入 `localStorage`
   - `Topbar` 切换语言后写入 `localStorage`
3. 即时渲染更新
   - 语言切换后当前页面文案立即变更
4. 本地格式化
   - 语言切换后时间格式随 locale 改变

### 11.2 页面覆盖测试

至少覆盖以下页面/组件：

- 登录页
- `Topbar`
- `SidebarNav`
- `Dashboard`

原因是这些位置基本覆盖了：

- 未登录入口
- 全局壳层
- 导航元信息
- 业务页中的动态文案与时间格式化

## 12. 迁移顺序

建议按以下顺序实施，减少回归风险：

1. 接入 `react-intl` 与 locale store，完成 provider 基线
2. 新增语言消息词典与 locale 检测逻辑
3. 先改登录页与 `Topbar`，让语言切换主路径成立
4. 改造 `navigation.ts`、`SidebarNav` 与页面标题渲染
5. 迁移 `Dashboard` 等页面中的静态文案、动态文案和时间格式化
6. 补齐国际化相关单元测试

这种顺序可以让“基础设施 -> 切换入口 -> 全局框架 -> 业务页 -> 测试”的路径逐步收敛，出现问题时也更容易定位。

## 13. 风险与控制

### 13.1 风险：页面中残留英文文案

如果只迁移部分页面或只改主路径，容易出现“框架切成中文，但某些卡片、弹窗、错误提示仍是英文”的半成品状态。

控制方式：

- 在本轮把现有页面的用户可见静态文案尽量一次性收口进词典
- 通过测试覆盖登录页、Topbar、Sidebar 和 Dashboard 主路径

### 13.2 风险：语言状态与登录状态耦合

如果把语言选择存进 auth store 或在登出时一起清空，用户会频繁丢失语言偏好。

控制方式：

- 为 locale 保持独立 store 和独立本地存储 key
- 明确语言偏好不参与登录态清理流程

### 13.3 风险：导航与页面标题定义重复

如果导航和页面标题分别维护翻译文本，后续会发生不同步。

控制方式：

- 让 `navigation.ts` 成为标题与描述的单一来源
- 页面壳层只消费 message id，不自行复制路由元信息

### 13.4 风险：时间格式仍跟浏览器默认走

如果只翻译文本、不改格式化逻辑，切换语言后时间格式可能与界面语言不一致。

控制方式：

- 统一收口现有 `toLocaleString()` 调用
- 通过 locale 上下文做显式格式化

## 14. 验收标准

完成后应满足以下验收标准：

1. `web/` 支持 `zh-CN` 与 `en`
2. 初次进入页面时，默认按浏览器语言决定界面语言
3. 用户手动切换语言后，刷新页面仍保留该选择
4. 登录页与登录后的 `Topbar` 都能切换语言
5. 语言切换后，页面文案无需刷新即可即时更新
6. `SidebarNav`、`Topbar`、登录页、Dashboard 至少全部可正确中英文切换
7. 前端生成错误文案和时间格式化跟随当前 locale
8. 新增语言时，只需增加消息词典与必要映射，而不需要重写页面逻辑

## 15. 实施结论

本轮采用 `react-intl` 作为 `web/` 的国际化基础设施，建立独立的 locale store、集中词典和 provider 装配方式，先支持 `zh-CN` 与 `en`，并在登录页与 `Topbar` 提供显式切换入口。

该方案兼顾了当前落地成本与后续扩展性，能够在不引入过重架构的前提下，把管理台从“英文写死页面”演进为“可持续扩语言的正式国际化前端”。

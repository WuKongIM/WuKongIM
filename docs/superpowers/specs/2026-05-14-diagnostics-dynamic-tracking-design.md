# Diagnostics Dynamic Tracking Design

## Goal

Allow operators to use the `web/` manager diagnostics page to temporarily retain message diagnostics for either one channel or one sending UID.

A "user tracking" rule means sender-side tracking only: messages whose `FromUID` equals the configured UID. It does not track messages delivered to that UID.

## Context

The current message diagnostics feature already supports querying retained events through manager APIs:

- `web/src/pages/diagnostics/page.tsx` can query by trace id, client message number, channel plus sequence, and recent errors.
- `internal/access/manager/diagnostics.go` exposes `/manager/diagnostics/*` read APIs under `cluster.diagnostics:r`.
- `internal/usecase/management/diagnostics.go` aggregates node-local diagnostics results across cluster nodes.
- `internal/observability/diagnostics` owns bounded in-memory event retention, indexes, redaction, and sampling.
- `WK_DIAGNOSTICS_DEBUG_MATCHES` already supports static high-priority sampling matches, but only from startup config.

Operators need a runtime control from the manager UI. Editing `wukongim.conf` and restarting a node is too slow for incident diagnosis and is unsafe for short-lived high-volume sampling.

## Scope

In scope:

- Runtime, TTL-bound diagnostics tracking rules for channels and sending UIDs.
- Manager APIs to create, list, and delete tracking rules.
- Node RPC so manager can apply rules to all eligible cluster nodes.
- Web UI controls on the existing diagnostics page.
- Sender UID propagation into diagnostics sampling where the send path already knows `FromUID`.
- Query support for recent events by `uid` and by `channel_key`.
- Permission split between diagnostics read and diagnostics tracking writes.

Out of scope:

- Tracking messages delivered to a UID.
- Persisting tracking rules in controller metadata or disk state.
- Replaying historical events that sampling already dropped.
- External tracing systems such as OpenTelemetry or Jaeger.
- Payload inspection, payload search, or response exposure of unredacted UID fields.
- Long-running rules without TTL.
- New standalone deployment semantics; a single node remains a single-node cluster.

## Approach

Use dynamic in-memory tracking rules, distributed by manager to each eligible node. The diagnostics sampler checks dynamic rules before normal sampling.

This is preferred over reusing `WK_DIAGNOSTICS_DEBUG_MATCHES` directly because startup config cannot support temporary UI-driven diagnosis. It is also preferred over controller-persisted rules for the first slice because diagnostics tracking is operational and short-lived; persistence raises cleanup and safety risks.

## User Semantics

### Channel Tracking

The operator enters:

- `channel_id`
- `channel_type`
- TTL
- sample rate, default `1.0`

The backend converts this to the diagnostics-safe channel key format used by existing sendtrace events:

```text
channel/<channel_type>/<base64url(channel_id without padding)>
```

The UI should not ask operators to hand-type `channel_key` for rule creation.

### Sender UID Tracking

The operator enters:

- `uid`
- TTL
- sample rate, default `1.0`

The rule matches events whose internal diagnostics `FromUID` equals that UID. Events returned to manager/web remain redacted and do not expose `from_uid` in event rows.

Sender UID tracking only guarantees future events after the rule is installed. It cannot recover events dropped before the rule existed.

## Backend Design

### Diagnostics Rule Model

Add a runtime diagnostics tracking rule model under `internal/observability/diagnostics`:

```go
type TrackingTarget string

const (
    TrackingTargetChannel   TrackingTarget = "channel"
    TrackingTargetSenderUID TrackingTarget = "sender_uid"
)

type TrackingRule struct {
    // ID is generated by manager and shared by every node that receives the rule.
    ID string
    // Target identifies whether the rule matches a channel or a sending UID.
    Target TrackingTarget
    // UID matches Event.FromUID when Target is sender_uid.
    UID string
    // ChannelKey matches Event.ChannelKey when Target is channel.
    ChannelKey string
    // SampleRate is the keep probability applied when this rule matches.
    SampleRate float64
    // CreatedAt records the node-local installation time.
    CreatedAt time.Time
    // ExpiresAt records when the node-local rule becomes inactive.
    ExpiresAt time.Time
}
```

Rules are node-local, bounded, and temporary:

- Default maximum active manual rules: 100.
- Default maximum TTL: 24 hours.
- Minimum valid TTL: 1 second.
- `sample_rate` must be between `0` and `1`.
- Expired rules are ignored on the hot path and pruned during list/create/delete operations.
- Duplicate `rule_id` updates the existing rule, making manager retries idempotent.

These limits can start as package constants. Do not add config fields unless operational feedback requires tuning.

### Sampler Changes

Extend `Sampler` so it can consult dynamic rules before existing debug/static sampling:

1. Dynamic tracking rules.
2. Existing static `DebugMatches` from `WK_DIAGNOSTICS_DEBUG_MATCHES`.
3. Slow threshold.
4. Error sample rate.
5. Baseline sample rate.

A dynamic rule hit returns `sample_reason=debug` to preserve current UI semantics.

The static `WK_DIAGNOSTICS_DEBUG_MATCHES` path remains supported. After `FromUID` propagation is added, its existing `uid` field becomes effective for sender-side matching.

### Event And Query Fields

Extend `diagnostics.Event` and `diagnostics.Query` with sender UID support:

```go
type Event struct {
    // FromUID is the sending UID used only for local matching and must be redacted before responses.
    FromUID string `json:"from_uid,omitempty"`
}

type Query struct {
    // UID filters local retained events by sender UID.
    UID string `json:"uid,omitempty"`
}
```

Store behavior:

- Add a bounded `uid` index.
- `matchesQuery` filters `q.UID` against internal `event.FromUID`.
- `redactEvent` continues clearing `FromUID` before any result leaves the store.
- Node RPC query encoding includes `UID` because remote nodes need it for filtering.
- Event RPC encoding does not need to carry `FromUID` if all store query outputs are already redacted.

Manager DTO behavior:

- `DiagnosticsQueryDTO` may echo `uid` because the operator supplied it as a query selector.
- `DiagnosticsEventDTO` must not include `from_uid`.

### Sendtrace Sender Propagation

Add `FromUID` to `pkg/observability/sendtrace.Event`:

```go
type Event struct {
    // FromUID is the sending UID used for diagnostics sampling and is not exposed in manager event DTOs.
    FromUID string
}
```

Populate it where the sending UID is already known:

- `internal/access/gateway/frame_router.go`: gateway send and sendack events use `cmd.FromUID`.
- `internal/usecase/message/send.go`: durable send event uses `cmd.FromUID`.
- `internal/app/channelcluster.go`: local and forwarded channel append events use `req.Message.FromUID`.

For leader replica events that currently only see `channel.Record`, add diagnostics-only metadata to `channel.Record` only if needed to retain deeper sender-scoped stages:

```go
type Record struct {
    // TraceID carries diagnostics correlation metadata and is not persisted.
    TraceID string
    // ClientMsgNo carries diagnostics correlation metadata and is not persisted.
    ClientMsgNo string
    // FromUID carries diagnostics sender metadata and is not persisted or exposed.
    FromUID string
}
```

`pkg/channel/handler.Append` can set these fields from the draft message before calling the replica group. Replica range events should only copy these fields when the range belongs to one known record, or when every record in the range has the same value, to avoid lying about mixed batches.

Follower fetch/apply events do not need to decode payloads for UID in the first slice. Channel tracking still retains those events by `channel_key`; sender UID tracking focuses on sender-known stages and any exact metadata propagated through the leader path.

## Manager API

Add routes under `/manager`:

```text
GET    /manager/diagnostics/tracking-rules
POST   /manager/diagnostics/tracking-rules
DELETE /manager/diagnostics/tracking-rules/:rule_id
```

Extend the existing events query:

```text
GET /manager/diagnostics/events?uid=<sender_uid>&channel_key=<channel_key>&stage=<stage>&result=<result>&node_id=<node_id>&limit=<limit>
```

`uid` and `channel_key` can both be accepted and interpreted as an AND filter, but the web UI should normally send only one based on the selected rule.

### Create Request

Sender UID rule:

```json
{
  "target": "sender_uid",
  "uid": "u1",
  "ttl_seconds": 3600,
  "sample_rate": 1.0
}
```

Channel rule:

```json
{
  "target": "channel",
  "channel_id": "g1",
  "channel_type": 2,
  "ttl_seconds": 3600,
  "sample_rate": 1.0
}
```

Validation:

- `target` must be `channel` or `sender_uid`.
- `sender_uid` requires non-empty trimmed `uid`.
- `channel` requires non-empty trimmed `channel_id` and positive `channel_type`.
- `ttl_seconds` must be positive and must not exceed the maximum TTL.
- `sample_rate` defaults to `1.0` when omitted and must be in `[0, 1]`.
- Invalid requests return `400 bad_request`.

### Response Shape

Create response:

```json
{
  "status": "ok",
  "rule": {
    "rule_id": "diag_rule_abc",
    "target": "sender_uid",
    "uid": "u1",
    "channel_key": "",
    "sample_rate": 1.0,
    "created_at": "2026-05-14T10:00:00Z",
    "expires_at": "2026-05-14T11:00:00Z"
  },
  "nodes": [
    {"node_id": 1, "status": "ok", "notes": []},
    {"node_id": 2, "status": "ok", "notes": []}
  ],
  "notes": []
}
```

List response:

```json
{
  "status": "ok",
  "rules": [],
  "nodes": [],
  "notes": []
}
```

Delete response:

```json
{
  "status": "ok",
  "rule_id": "diag_rule_abc",
  "nodes": [],
  "notes": []
}
```

Top-level status values:

- `ok`: every targeted eligible node accepted the operation.
- `partial`: at least one eligible node failed or was skipped, but at least one node succeeded.
- `error`: all eligible nodes failed.

Dead nodes are skipped using the same cluster targeting semantics as diagnostics query aggregation.

## Permissions

Use the existing manager permission model:

- Read diagnostics and list tracking rules: `cluster.diagnostics:r`.
- Create/delete tracking rules: `cluster.diagnostics:w`.

Update default permission resource descriptions so admins can grant write access separately from read-only diagnostics users.

## Management Usecase

Add narrow management methods instead of putting business logic in access handlers:

```go
type DiagnosticsTrackingRequest struct {
    Target      string
    UID         string
    ChannelID   string
    ChannelType uint8
    TTL         time.Duration
    SampleRate  float64
}

type DiagnosticsTrackingNode interface {
    AddDiagnosticsTrackingRule(ctx context.Context, nodeID uint64, rule diagnostics.TrackingRuleInput) (diagnostics.TrackingRule, error)
    ListDiagnosticsTrackingRules(ctx context.Context, nodeID uint64) ([]diagnostics.TrackingRule, error)
    DeleteDiagnosticsTrackingRule(ctx context.Context, nodeID uint64, ruleID string) error
}
```

The management app should:

- Normalize and validate manager requests.
- Generate one `rule_id` per create request.
- Convert `channel_id + channel_type` to `channel_key` once in the usecase.
- Resolve eligible nodes exactly like diagnostics query aggregation.
- Apply node operations concurrently with a bounded semaphore.
- Return per-node status and aggregate notes.
- Avoid a special non-cluster branch for single-node deployment.

`internal/app` wires local calls directly for the current node and node RPC for remote nodes, matching the existing `managementDiagnosticsReader` pattern.

## Node RPC

Add node RPC services for:

- Add tracking rule.
- List tracking rules.
- Delete tracking rule.

The codec must include:

- `rule_id`
- `target`
- `uid`
- `channel_key`
- `ttl_seconds` or `expires_at`
- `sample_rate`

Use idempotent add/update behavior so manager retries do not create duplicate rules.

## Web Design

Extend `web/src/pages/diagnostics/page.tsx` with a `Tracking Rules` section above or beside the current query form.

Create form:

- Target select: `Sender UID` / `Channel`.
- Sender UID input when target is sender.
- `Channel ID` and `Channel Type` inputs when target is channel.
- TTL preset select: `10m`, `1h`, `6h`, custom seconds.
- Sample rate input defaulted to `1`.
- Submit button: `Start tracking`.

Rule list:

- Target summary.
- Expiration time.
- Sample rate.
- Aggregate node status.
- Actions:
  - `Query recent events`.
  - `Stop tracking`.

`Query recent events` behavior:

- Sender UID rule calls `getDiagnosticsEvents({ uid, limit })`.
- Channel rule calls `getDiagnosticsEvents({ channelKey, limit })`.
- The existing timeline, node results, event table, related links, and export JSON sections render the returned response.

Web files:

- `web/src/lib/manager-api.types.ts`: tracking rule request/response types, `DiagnosticsEventsParams.uid`, `DiagnosticsEventsParams.channelKey`.
- `web/src/lib/manager-api.ts`: list/create/delete rule clients and extended events query builder.
- `web/src/pages/diagnostics/page.tsx`: UI state and actions.
- `web/src/pages/diagnostics/page.test.tsx`: form validation, create, delete, query from rule, 403 handling.
- `web/src/i18n/messages/en.ts` and `web/src/i18n/messages/zh-CN.ts`: new labels and validation messages.

## Safety And Privacy

- Dynamic tracking rules are temporary and must require TTL.
- Rule count and TTL are bounded to prevent accidental unbounded sampling.
- No message payloads are recorded or exposed.
- `FromUID` is used for local matching and querying but is redacted from event DTOs.
- Manager write operations require `cluster.diagnostics:w`.
- Existing diagnostics buffer capacity still bounds total retained events.
- High-volume channel rules can increase buffer churn; the UI should display a short warning that tracking only affects future retained events and may evict older diagnostics sooner.

## Error Handling

Backend:

- Invalid JSON or target fields: `400 bad_request`.
- Missing management dependency: `503 service_unavailable`.
- Missing permission: existing manager auth `403`.
- Node RPC timeout or unavailable node: per-node `unavailable`, aggregate `partial` when at least one node succeeded.
- All nodes unavailable: aggregate `error`.

Web:

- 403 shows a permission message mentioning `cluster.diagnostics:w` for create/delete.
- 503 shows diagnostics unavailable.
- Partial rule operations show node notes but still render successful nodes.
- Validation errors stay inline in the Tracking Rules card.

description: 说明消息事件流在离线读取时，如何按确定性规则合并成 MessageLaneState 投影。
---

这一页解释的不是“事件怎么发”，而是：

**一条消息下的多条事件，最终怎么稳定地合并成客户端可读的结果。**

在这套设计里，离线读取优先使用 `MessageLaneState` 投影，而不是每次都重放全部原始事件。

## 合并目标

合并规则围绕 3 个约束展开：

1. `Message` 是地基，`Event` 必须挂在 `client_msg_no` 上。
2. 合并在 `client_msg_no + lane_id` 维度执行。
3. 输出结果是 `MessageLaneState`，供 `messagesync` 等离线接口直接读取。

## 输入与输出

输入是某条消息下的一组事件：

- `client_msg_no`
- `lane_id`
- `msg_event_seq`
- `event_type`
- `payload`
- `visibility`
- `occurred_at`

输出是单个轨道的投影：

- `client_msg_no`
- `lane_id`
- `status`
- `last_msg_event_seq`
- `snapshot_payload`
- `end_reason`
- `error`

## 排序与幂等

合并时要先固定两个基础规则。

### 排序规则

唯一排序键是 `msg_event_seq`，按升序执行。

`occurred_at` 只用于展示，不参与合并顺序判断。

### 幂等规则

建议幂等键使用：

`(client_msg_no, event_id)`

如果同一事件重复到达，直接忽略。

### 序号空间规则

- `msg_event_seq` 是消息级事件序号
- 同一消息下不同 `lane_id` 共享同一序号空间
- 因此带 `lane_id` 过滤读取时，序号允许出现跳跃

## 通用状态机

支持的事件类型：

1. `stream.open`
2. `stream.delta`
3. `stream.snapshot`
4. `stream.close`
5. `stream.error`
6. `stream.cancel`

对应状态转换：

| 事件 | 状态变化 |
| --- | --- |
| `stream.open` | `open` |
| `stream.delta` | 只更新内容 |
| `stream.snapshot` | 只更新内容 |
| `stream.close` | `closed` |
| `stream.error` | `error` |
| `stream.cancel` | `cancelled` |

一旦进入终态：

- `closed`
- `error`
- `cancelled`

后续非幂等重复事件默认忽略，最多记录告警。

## 不同 `payload.kind` 的合并策略

### `kind=text`

- `stream.delta`：按顺序追加字符串
- `stream.snapshot`：直接覆盖当前文本快照
- `stream.close`：冻结最终文本

### `kind=tool`

- `stream.open`：初始化工具上下文，比如 `tool_name`、`tool_call_id`
- `stream.delta`：通常用于 `args_delta` 一类增量
- `stream.snapshot`：覆盖结果对象
- `stream.close`：冻结结果

### `kind=workflow` / `kind=order`

这一类更适合以 `stream.snapshot` 为主。

原因很简单：它们表达的重点通常不是“字符增量”，而是“当前状态是什么”。

### `kind=binary`

- `stream.delta`：按分片拼接，建议带 `chunk_index`
- `stream.snapshot`：可以直接覆盖成完整对象引用，比如对象存储 URL

## 合并伪代码

```text
for each lane in groupBy(client_msg_no, lane_id):
  state = loadOrInit(lane)
  events = sortBy(msg_event_seq)

  for e in events:
    if isDuplicate(client_msg_no, e.event_id):
      continue
    if isTerminal(state.status):
      continue

    switch e.event_type:
      case stream.open:
        state.status = open
        applyOpenMeta(state, e.payload)
      case stream.delta:
        applyDelta(state, e.payload.kind, e.payload)
      case stream.snapshot:
        applySnapshot(state, e.payload.kind, e.payload)
      case stream.close:
        state.status = closed
        applyClose(state, e.payload)
      case stream.error:
        state.status = error
        state.error = e.payload.error
      case stream.cancel:
        state.status = cancelled

    state.last_msg_event_seq = e.msg_event_seq

  saveLaneState(state)
```

## 离线接口怎么消费这个结果

### `POST /channel/messagesync`

这里不建议扫描所有原始事件。

更直接的做法是：

1. 读取消息对应的 `MessageLaneState`
2. 组装 `event_meta.lanes[]`
3. 继续兼容旧字段

旧字段的映射建议：

- `stream_data`：来自 `lane_id=main` 的聚合快照
- `end/end_reason/error`：来自 `lane_id=main` 的终态信息

### `POST /message/eventsync`

这个接口负责按 `client_msg_no + from_msg_event_seq` 返回增量事件，用于更细粒度的回放或补拉。

## 两个典型结果

### 文本单轨道

输入事件：

1. `stream.open`
2. `stream.delta` `{"kind":"text","delta":"你好"}`
3. `stream.delta` `{"kind":"text","delta":"，世界"}`
4. `stream.close`

合并结果：

```json
{
  "lane_id": "main",
  "status": "closed",
  "last_msg_event_seq": 4,
  "snapshot_payload": {
    "kind": "text",
    "text": "你好，世界"
  }
}
```

### 同消息多轨道

当一条消息同时有 `main` 和 `tool_weather` 两条轨道时，`messagesync` 可以返回：

```json
{
  "client_msg_no": "cmn_agent_001",
  "event_meta": {
    "has_events": true,
    "lane_count": 2,
    "open_lane_count": 1,
    "lanes": [
      {
        "lane_id": "main",
        "status": "open",
        "last_msg_event_seq": 12,
        "snapshot": {
          "kind": "text",
          "text": "北京今天 26°C，晴。"
        }
      },
      {
        "lane_id": "tool_weather",
        "status": "closed",
        "last_msg_event_seq": 8,
        "snapshot": {
          "kind": "tool",
          "tool_name": "weather.search",
          "tool_call_id": "tc_weather_01",
          "result": {
            "temp_c": 26,
            "condition": "Sunny"
          }
        }
      }
    ]
  }
}
```

## 实现建议

原始设计里最重要的实现倾向有 3 点：

1. 写路径里原子更新 `MessageLaneState`
2. `messagesync` 默认读投影，不扫全量事件
3. 终态轨道优先保留快照，而不是长期保存细粒度中间态

如果你的目标是先把 AI Agent 场景跑通，这 3 条的优先级通常比“先建一套完美的事件存储”更高。

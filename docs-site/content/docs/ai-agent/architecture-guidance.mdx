这一页不再重复协议字段，而是直接回答一个更实际的问题：

**如果你真的要把 AI Agent 事件流落到现有消息体系里，推荐怎么接。**

## 先定清楚三层职责

这套能力建议始终按 3 层理解：

1. `Message`
   负责作为锚点消息存在，承接 `message_id`、`client_msg_no`、`message_seq` 这些现有能力。
2. `MessageEvent`
   负责表达 AI 运行过程中的增量，例如文本 delta、工具结果、状态变化。
3. `MessageLaneState`
   负责离线可读的聚合结果，让客户端不必在补拉时每次重放全部事件。

这个分层的价值在于：

- 不破坏现有消息主链路
- 不把 Agent 中间态硬塞进基础消息 payload
- 不让离线读取依赖高成本的全量回放

## 推荐写路径

推荐写路径尽量保持短而稳定：

1. 先确保锚点消息已经存在。
2. 并且这条锚点消息创建时已经设置 `is_stream = 1`。
3. 通过 `client_msg_no` 追加事件。
4. 在同一条写链路里更新对应 `lane_id` 的投影状态。

更具体地说：

- 不要让客户端先依赖 `message_id` 再追加事件。
- 不要把普通消息直接当成事件锚点使用；锚点消息必须显式开启 `is_stream = 1`。
- 不要把 `lane_id` 当成消息主键。
- 不要把“事件追加成功”和“投影状态更新成功”拆成互相独立、无一致性保障的两段流程。

如果你的系统已经有 raft 提案或统一写路径，比较稳妥的做法是让事件处理和投影更新一起进入原子提交。

## 推荐读路径

在线和离线，建议分开思考。

### 在线场景

在线 UI 如果需要逐步展示生成过程，可以消费事件增量：

- `stream.delta`
- `stream.snapshot`
- `stream.close`

这里关注的是“过程”。

### 离线场景

离线补拉不要默认重放所有事件，而是优先走：

- `POST /channel/messagesync`
- 读取 `event_meta`
- 必要时再调 `POST /message/eventsync`

这里关注的是“恢复结果”。

这两种路径最好同时存在：

- 在线时保证体验细腻
- 离线时保证恢复高效

## 客户端接入顺序

如果你是第一次做这套能力，建议不要一上来就把所有 lane 和回放能力全做完。

更稳的顺序通常是：

1. 先让客户端在消息列表和消息详情里能读懂 `event_meta`
2. 先支持 `main` 轨道的聚合展示
3. 再补 `eventsync` 的细粒度回放
4. 最后再扩展多 lane，比如工具流和 workflow 流

这个顺序的好处是：

- 你会先拿到“离线可恢复”的主能力
- UI 不会因为追求全量回放而过早复杂化
- 后续扩展工具流时，也不会推翻基础消息展示逻辑

## 灰度与兼容建议

这套能力最适合渐进接入，而不是一次性替换所有消息读写逻辑。

推荐方式：

1. 保持老客户端继续读普通 `messagesync`
2. 通过开关或版本字段让新客户端识别 `event_meta`
3. 只在 AI Agent 场景消息上启用事件流扩展

这样做的原因很现实：

- 旧消息体系不需要重构
- 非 Agent 场景不必承担额外复杂度
- 兼容性问题会被收敛在一小段业务范围内

## 推荐的系统边界

如果要避免后面越做越乱，下面这几个边界尽量不要打破。

### 1. Agent 语义不要写死在 `event_type`

`event_type` 最好只表达通用协议动作，比如：

- `stream.open`
- `stream.delta`
- `stream.snapshot`
- `stream.close`

真正的业务差异放在 `payload.kind` 里。

### 2. `lane_id` 只用于消息内并行流

`lane_id` 的职责是隔离同一条消息下的多路事件，不是替代消息主键。

### 3. 离线读路径不要依赖总是回放全量事件

如果离线恢复必须每次扫描整条消息的事件历史，随着 Agent 使用变多，成本会迅速失控。

## 常见误区

下面这些做法在设计上最容易把系统带偏。

### 把 Agent 中间态直接塞进消息 payload

这样做短期省事，但很快会把“最终消息内容”和“运行过程事件”混成一团。

### 为每一种 Agent 能力发明一个新的事件类型

如果事件类型被业务语义占满，协议层很快会变得不可维护。

### 让客户端必须先拿到 `message_id`

这会把前后端流程绑死，破坏 `client_msg_no` 作为客户端主关联键的价值。

### 一开始就追求完整多轨道回放

大多数产品先需要的是“能稳定恢复结果”，不是“能百分之百复刻实时过程”。

## 一个务实的落地顺序

如果你要在工程里排优先级，可以按这个顺序推进：

1. 保留现有消息主链路不动
2. 增加 `POST /message/event` 写入口
3. 增加 `MessageLaneState` 投影更新
4. 扩展 `messagesync` 返回 `event_meta`
5. 最后再补 `eventsync` 和更复杂的多 lane 体验

这个顺序的核心思路是：

先把**可写**和**可恢复**打通，再去补更细的体验层能力。

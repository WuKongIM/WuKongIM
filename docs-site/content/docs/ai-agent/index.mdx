这一组文档主要回答一个问题：

如果你的业务不是普通 IM 消息，而是 **AI Agent 在一条消息下持续产出事件**，WuKongIM 应该怎么承载？

这里的重点场景包括：

- 大模型逐步输出文本
- 工具调用请求与结果回传
- Workflow / Order 一类状态推进事件
- 客户端离线后，通过消息主入口重新拿到聚合状态

## 这组文档和普通消息接口是什么关系

AI Agent 场景并没有脱离 WuKongIM 现有消息体系。

相反，这套设计的核心约束是：

- **消息是地基**
- **事件必须挂在某条消息上**
- **离线仍然优先通过 `POST /channel/messagesync` 读取**

也就是说，它是建立在现有消息模型上的增强能力，而不是另一套完全独立的协议栈。

- 做 AI 助手、Copilot、智能客服的服务端团队
- 需要把文本流、工具流、状态流统一落到消息体系里的平台团队

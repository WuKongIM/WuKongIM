`消息事件流（Message-Centric Event Stream）` 的核心思想很简单：

**消息是地基，所有事件都必须关联到某条消息。**

这套设计虽然主要面向 AI Agent，但协议本身保持通用，不绑定具体 Agent 框架。

## 设计目标

这套模型主要想同时满足 4 件事：

1. 保持现有 `POST /channel/messagesync` 作为离线主入口，不破坏已有客户端。
2. 支持一条消息下的多路并行流，比如文本流、工具流、状态流。
3. 发送事件时只依赖 `client_msg_no`，不要求客户端提前拿到 `message_id`。
4. 事件协议保持通用，不把语义写死在 Agent 专属字段里。

## 核心模型

### Message

沿用现有消息模型：

- `message_id`
- `client_msg_no`
- `message_seq`

其中 `client_msg_no` 继续作为客户端侧最重要的关联键。

如果某条消息后续还要通过事件流继续追加内容，这条锚点消息必须是流式消息，也就是创建时要带：

```json
{
  "is_stream": 1
}
```

这是事件流成立的前置约束，不只是“可选优化项”。

### MessageEvent

事件是消息的子资源，建议唯一性使用：

`(channel_id, channel_type, client_msg_no, event_id)`

推荐字段如下：

| 字段 | 说明 |
| --- | --- |
| `client_msg_no` | 消息关联键，必填 |
| `lane_id` | 消息内事件轨道，默认 `main` |
| `event_id` | 幂等键，必填 |
| `event_type` | 通用事件类型 |
| `payload` | 结构化 JSON 负载 |
| `visibility` | `public` / `private` / `restricted` |
| `occurred_at` | 事件发生时间，毫秒 |
| `msg_event_seq` | 服务端分配的消息级递增序号 |

需要注意的点：

- `msg_event_seq` 的作用域是同一条消息。
- 不同 `lane_id` 共享同一序号空间，不会按 lane 单独重置。

### MessageLaneState

这是给离线读取准备的投影层，不要求客户端离线后把所有事件逐条重放一遍。

推荐字段：

- `client_msg_no`
- `lane_id`
- `status`
- `last_msg_event_seq`
- `snapshot_payload`
- `end_reason`
- `error`

## 通用事件类型

推荐保留一组通用事件语义：

1. `stream.open`
2. `stream.delta`
3. `stream.snapshot`
4. `stream.close`
5. `stream.error`
6. `stream.cancel`

业务差异不通过 `event_type` 扩散，而是通过 `payload.kind` 区分，例如：

- `text`
- `tool`
- `workflow`
- `order`
- `asr`

## 三个核心接口

### 1. 发送事件：`POST /message/event`

这个接口用于向某条消息追加事件。客户端只传 `client_msg_no`，不传 `message_id`。

但要注意，能被追加事件的前提是：

- 这条锚点消息已经存在
- 并且创建锚点消息时已经设置 `is_stream = 1`

请求示例：

```json
{
  "channel_id": "u2",
  "channel_type": 1,
  "from_uid": "u1",
  "client_msg_no": "cmn_abc_001",
  "event_id": "evt_0001",
  "event_type": "stream.delta",
  "lane_id": "main",
  "visibility": "public",
  "occurred_at": 1772860800000,
  "payload": {
    "kind": "text",
    "delta": "你好，"
  },
  "headers": {
    "schema": "v1",
    "source": "llm-a"
  }
}
```

服务端处理规则：

1. 通过 `(channel_id, channel_type, client_msg_no)` 找到消息。
2. 找不到时，按配置选择创建锚点消息或直接报错。
3. 如果消息不是流式消息（`is_stream != 1`），应直接拒绝事件追加。
4. 追加事件并分配 `msg_event_seq`。
5. 同步更新 `MessageLaneState`。

响应示例：

```json
{
  "status": 200,
  "data": {
    "client_msg_no": "cmn_abc_001",
    "lane_id": "main",
    "event_id": "evt_0001",
    "msg_event_seq": 12,
    "stream_status": "open",
    "message_seq": 10241
  }
}
```

### 2. 离线消息主入口扩展：`POST /channel/messagesync`

这不是新增一个平行接口，而是在原有离线消息同步能力上扩展 `event_meta`。

请求示例：

```json
{
  "login_uid": "u1",
  "channel_id": "u2",
  "channel_type": 1,
  "start_message_seq": 10000,
  "end_message_seq": 0,
  "limit": 100,
  "pull_mode": 1,
  "stream_v2": 1,
  "include_event_meta": 1,
  "event_summary_mode": "basic"
}
```

单条消息的返回示意：

```json
{
  "message_id": 73918291023123,
  "message_seq": 10241,
  "client_msg_no": "cmn_abc_001",
  "payload": "base-message-payload",
  "stream_data": "aggregated-bytes",
  "event_meta": {
    "has_events": true,
    "event_version": 7,
    "last_msg_event_seq": 12,
    "lane_count": 2,
    "open_lane_count": 1,
    "lanes": [
      {
        "lane_id": "main",
        "status": "open",
        "last_msg_event_seq": 12,
        "snapshot": { "text": "你好，今天北京天气..." }
      },
      {
        "lane_id": "tool_weather",
        "status": "closed",
        "last_msg_event_seq": 5,
        "end_reason": 0
      }
    ]
  },
  "event_sync_hint": {
    "client_msg_no": "cmn_abc_001",
    "from_msg_event_seq": 0
  }
}
```

它的价值是：

- 老客户端仍然能沿用消息同步入口
- 新客户端可以识别 `event_meta`
- 需要细粒度回放时，再去补拉事件增量

### 3. 事件增量同步：`POST /message/eventsync`

这个接口用于按消息补拉更细粒度的事件。

需要记住的游标语义：

- `from_msg_event_seq` / `next_msg_event_seq` 是**消息级游标**
- 如果带 `lane_id` 过滤，返回序号可能不连续，因为不同 lane 共享同一序号空间

请求示例：

```json
{
  "channel_id": "u2",
  "channel_type": 1,
  "client_msg_no": "cmn_abc_001",
  "lane_id": "",
  "from_msg_event_seq": 0,
  "limit": 200,
  "include_private": 0
}
```

响应示例：

```json
{
  "client_msg_no": "cmn_abc_001",
  "from_msg_event_seq": 0,
  "next_msg_event_seq": 200,
  "more": 1,
  "events": [
    {
      "msg_event_seq": 1,
      "event_id": "evt_0001",
      "lane_id": "main",
      "event_type": "stream.open",
      "visibility": "public",
      "occurred_at": 1772860800000,
      "payload": { "kind": "text" }
    }
  ]
}
```

## `lane_id` 是做什么的

`lane_id` 不参与消息定位。消息定位仍然只看 `client_msg_no`。

它的作用是把同一条消息内的并行事件流隔离开，例如：

- `main`：文本输出
- `tool_weather`：工具调用
- `workflow`：状态推进

单流业务里可以不传，默认使用 `main`。

## 三个典型场景

### 文本流

最简单的序列通常是：

1. `stream.open`
2. 多次 `stream.delta`
3. `stream.close`

但在这之前，必须先有一条 `is_stream = 1` 的锚点消息。

离线时，客户端先从 `messagesync` 取聚合结果；只有需要逐字回放时，再补调 `eventsync`。

### 工具流

同一条消息下除了文本，还可以挂工具轨道：

- `main`：自然语言输出
- `tool_*`：工具参数、结果与结束状态

这样客户端既能展示文本，也能展示工具执行进度。

### 工作流状态流

如果消息代表一轮 AI 任务，`workflow` 轨道可以承载阶段状态、步骤推进或订单类状态变化，而不必把所有状态都硬编码进文本内容里。

## 和普通消息接口的关系

如果你先前只接过 [`服务端消息`](/server/message)，可以把这套能力理解成：

- 普通消息负责“消息本身”
- 事件流负责“消息运行期间不断追加的状态和内容”

两者不是竞争关系，而是分层关系。

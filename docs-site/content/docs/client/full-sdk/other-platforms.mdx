这一页处理的是这类平台：

- 能接入 WuKongIM
- 但当前不单独拆完整平台文档树
- 本质上复用 JavaScript / Web SDK 的连接和消息能力

最核心的判断只有一个：

**如果这个平台本质上复用 Web 运行时或 JavaScript SDK 能力，就先按 Web SDK 路径来理解。**

## 当前怎么理解这些平台

### 小程序

如果你正在做小程序接入，先不要纠结“有没有小程序专属文档树”。  
更重要的是先把完整链路理顺：

- 连接和鉴权链路怎么走
- 消息、会话、频道能力分别由谁负责
- 哪些能力依赖你自己的业务数据源

这些内容都在 Web SDK 文档路径里更完整。

### UniApp

如果你做的是 UniApp，也建议先按 Web SDK 的模型接起来，再根据运行环境差异补适配。

### 其他 JS 运行时

如果你的平台本质上仍然是 JavaScript 运行时，只是宿主环境不同，也建议先走 Web SDK 总览和集成页。

## 什么时候不应该直接照搬 Web SDK

下面这些情况要额外评估：

- 宿主环境对 WebSocket、存储、文件上传能力有限制
- 运行时不是标准浏览器，也不是标准 Node.js
- 页面生命周期和前端框架事件模型差异很大

这时最稳妥的做法是：

1. 先跑通最小连接和消息链路
2. 再补消息同步、会话同步、频道资料这些扩展能力
3. 最后处理平台特有的兼容问题

## 最推荐的阅读顺序

先把“完整客户端到底要负责什么”看清楚，再落到你的运行环境差异上，会省很多时间。

## 什么时候该停下来单独评估

如果你发现自己遇到的不是“文档没写到”，而是“宿主平台能力和 Web 不一样”，就不要继续照搬。  
这时应该优先验证：

- WebSocket 是否稳定可用
- 本地存储是否足够支撑消息和会话缓存
- 生命周期回调是否会影响连接重连和同步时机

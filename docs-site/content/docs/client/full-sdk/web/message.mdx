---
title: 消息
description: 说明 Web SDK 的消息发送、接收与同步能力。
---

Web SDK 的消息能力主要围绕 `chatManager` 展开。

你可以把它先理解成三个问题：

- 怎么发消息
- 怎么监听新消息和发送结果
- 怎么补拉历史消息

## 1. 发送消息

最常见的是先构造消息内容，再指定目标频道发送。

```ts
import {
  Channel,
  ChannelTypePerson,
  MessageText,
  WKSDK,
} from 'wukongimjssdk';

const to = new Channel('u10002', ChannelTypePerson);
const content = new MessageText('hello');

WKSDK.shared().chatManager.send(content, to);
```

群聊和单聊的主要差别不在发送方法，而在目标频道不同。

## 2. 监听收到的新消息

```ts
const onMessage = (message) => {
  console.log('new message', message);
};

WKSDK.shared().chatManager.addMessageListener(onMessage);
```

页面销毁或模块卸载时，记得配套移除监听。

## 3. 监听发送状态

发送动作发起后，不代表业务上已经成功确认。你通常还需要监听发送回执状态：

```ts
const onStatus = (ack) => {
  console.log(ack.reasonCode, ack.clientSeq);
};

WKSDK.shared().chatManager.addMessageStatusListener(onStatus);
```

这一步通常用来做：

- 发送中/成功/失败 UI
- 本地临时消息和服务端确认结果的对应
- 重试或错误提示

## 4. 历史消息同步不是 SDK 自己凭空拉的

这一点非常重要。

如果你希望 Web 端支持：

- 首次进入会话加载历史消息
- 上拉查看更多消息
- 重连后按需补拉

那就要先实现消息同步数据源：

```ts
WKSDK.shared().config.provider.syncMessagesCallback = async (
  channel,
  opts,
) => {
  return fetchYourMessages(channel, opts);
};
```

之后 SDK 才能在需要时调用这条链路。

## 5. 典型消息链路怎么串起来

一个常见顺序是：

1. 建立连接
2. 监听新消息和发送状态
3. 进入某个频道时加载最近一段消息
4. 用户继续上拉时按 `opts` 补拉更多历史消息

## 最容易踩的坑

- 只监听新消息，不处理发送状态，导致 UI 一直不知道是否发送成功
- 没实现 `syncMessagesCallback`，却希望 SDK 能自动补历史消息
- 把“消息列表”逻辑和“会话列表”逻辑混在一起
- 直接把所有历史消息一次性全拉下来，导致加载过重

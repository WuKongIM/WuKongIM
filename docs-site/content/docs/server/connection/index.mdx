---
title: 连接管理
description: 收纳连接移除、踢下线等连接管理能力。
---

这组接口解决的是“连接层管理”，不是消息层管理。

最常见的两个动作是：

- 主动移除某个连接
- 主动踢掉某个连接

除此之外，还有一个非常常用的排查入口：

- 查看当前连接信息

## 先分清三个接口

### 1. 查看连接信息

```text
GET /connz
```

这个接口更适合：

- 排查某个用户当前连在哪
- 看连接总数
- 看某个节点上的连接状态

常用查询参数包括：

- `offset`
- `limit`
- `uid`
- `node_id`
- `sort`

如果你是排查线上连接状态，通常先从这里开始，而不是先做踢连接。

### 2. 移除连接

```text
POST /conn/remove
```

它的作用更接近“直接关闭这个连接”。

最常见请求体字段：

- `uid`
- `conn_id`
- `node_id`
- `op_node_id`

### 3. 踢掉连接

```text
POST /conn/kick
```

和移除连接的区别是，它会先给客户端写一个断开连接通知，再在稍后关闭连接。

如果你希望客户端明确感知“被服务端踢下线”，优先看这个接口。

## `remove` 和 `kick` 怎么选

经验上可以直接这样理解：

- 只想关闭连接：`/conn/remove`
- 希望客户端收到明确踢下线语义：`/conn/kick`

## 集群场景下怎么理解 `node_id` / `op_node_id`

这两个字段的存在，是因为连接不一定在当前节点上。

也就是说：

- 你发请求的节点
- 真正持有连接的节点

可能不是同一个。

当前实现里，如果目标连接不在本节点，会按节点信息继续转发。

## 排查连接问题的推荐顺序

1. 先查 `/connz`
2. 确认目标 `uid`、`conn_id`、`node_id`
3. 再决定是 `remove` 还是 `kick`

## 常见坑

### 1. 不先查连接信息，直接踢

这样很容易把请求参数写错。

### 2. `conn_id` 传 0

当前实现会直接报错。

### 3. 把连接管理当成用户管理

连接是“某个用户的一次在线连接”，不是用户账号本身。

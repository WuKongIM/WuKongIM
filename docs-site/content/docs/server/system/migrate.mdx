核心接口：

```text
GET /migrate/result
```

这个接口的作用不是发起迁移，而是**查看迁移任务当前结果**。

如果你正在做老版本数据迁移，或者怀疑迁移没有完成，这个接口是第一观察入口。

## 什么时候用

- 部署后确认迁移是否已经完成
- 排查为什么服务还在迁移阶段
- 观察当前迁移卡在 `message`、`user`、`channel` 哪一步

## 最小示例

```bash
curl http://localhost:5001/migrate/result
```

## 典型返回

```json
{
  "status": "running",
  "step": "user",
  "last_err": null,
  "try_count": 2
}
```

## 返回字段怎么理解

| 字段 | 说明 |
| --- | --- |
| `status` | 迁移整体状态 |
| `step` | 当前或最后一次迁移步骤 |
| `last_err` | 最近一次错误 |
| `try_count` | 当前步骤已尝试次数 |

## `status` 常见值

- `running`：仍在执行迁移
- `completed`：当前任务已经完成并停止
- `migrated`：已经处于迁移完成后的稳定状态

## `step` 常见值

当前仓库里能看到的迁移步骤主要是：

- `message`
- `user`
- `channel`

如果接口一直停在某一步，通常说明该步骤还没走完，或者这一阶段在反复重试。

## 怎么排查这个结果

### 1. `status = running`

先看：

- `step` 卡在哪一段
- `try_count` 是否持续增加
- `last_err` 是否为空

### 2. `last_err` 不为空

这通常就是最直接的排查入口。

优先去服务日志里对照这条错误，而不是盲目重启。

### 3. `status = migrated`

说明当前数据目录已经被标记为迁移完成。

这时如果你还怀疑数据不完整，应该转去看实际消息、用户、频道数据，而不是继续轮询迁移结果接口。

## 常见坑

### 1. 以为这个接口会“触发迁移”

不会。它只是结果查看接口。

### 2. 一直轮询，但不看日志

`try_count` 和 `last_err` 只能告诉你方向，真正的细节还是要回到服务端日志。

### 3. 把 `completed` 和 `migrated` 理解成完全相同

这两个值都表示“不是正在运行中”，但语义上仍有区别。  
对运维来说，更重要的是确认当前是否还在跑迁移、有没有错误。

核心接口：

```text
GET /health
```

这是你最应该先接入的系统接口之一。

它的用途不是“看服务是否启动了”，而是更进一步判断：

- 当前节点 HTTP 服务是否可响应
- 当前集群状态是否正常

## 什么时候用它

- 容器或进程启动后的存活检查
- 反向代理或负载均衡健康检查
- 发布后的最小回归验证
- 日常监控和告警

## 最小示例

```bash
curl http://localhost:5001/health
```

## 正常返回

```json
{
  "status": "ok"
}
```

## 异常返回

如果集群状态检查失败，会返回：

```json
{
  "status": "error",
  "message": "..."
}
```

HTTP 状态码通常会是 `500`。

## 这个接口实际检查了什么

当前仓库实现里，`/health` 不只是简单返回固定字符串。

如果当前节点跑在分布式集群模式下，它还会调用集群状态检查逻辑；检查失败时直接返回错误。

所以它比“进程是否还活着”更有价值。

## 使用建议

### 1. 把它接到最外层入口

无论你是 Docker、Linux 直跑，还是网关层转发，都应该确保能快速访问到 `/health`。

### 2. 监控频率保持简单

通常每 30 到 60 秒检查一次就足够；对外探测过于频繁没有太大意义。

### 3. 不要把它和业务链路验证混为一谈

`/health` 正常只说明系统基础状态可用，不代表：

- 客户端一定能连上
- 路由地址一定配置正确
- 消息收发一定完全正常

这些还要配合：

- `/route`
- `/message/send`
- 客户端真实连接测试

## 常见坑

### 1. 只看进程，不看 `/health`

进程还在不等于服务可用，尤其是集群状态异常时更明显。

### 2. 只检查节点内网，不检查实际入口

如果你前面还有 Nginx 或网关，最好入口层和节点层都做检查。

### 3. 把 `/health` 当成完整验收

它只是第一步，不是全部验收。

## 命令行验证

如果你在用仓库自带 CLI，也可以直接执行健康检查命令：

```bash
go run main.go health
```

或者使用你已经构建好的 `wkcli` 对应命令。

会话接口解决的问题不是“消息怎么发”，而是：

- 用户当前有哪些最近会话
- 每个会话的未读怎么维护
- 会话删除、同步怎么做

如果你已经能收发消息，但客户端最近会话列表、未读数、删除状态还没对上，通常就要回到这里。

## 当前这组接口里最重要的能力

### 1. 同步会话

```text
POST /conversation/sync
```

这是会话能力里最重要的入口。

它适合：

- 客户端启动时拉最近会话
- 会话列表增量同步
- 获取每个会话的未读、最后消息位置、最近消息片段

常见请求字段包括：

- `uid`
- `version`
- `last_msg_seqs`
- `msg_count`
- `only_unread`
- `page`
- `page_size`

## 2. 同步会话最近消息

```text
POST /conversation/syncMessages
```

这个接口更适合：

- 会话列表首屏补最近几条消息
- 减少客户端自己逐会话补拉的次数

## 3. 通过频道集合同步会话

```text
POST /conversation/syncByChannels
```

如果你已经明确知道一批频道，想按频道维度回收会话信息，可以看这个接口。

## 4. 获取最近会话对应频道集合

```text
POST /conversation/channels
```

这个接口更像一个辅助接口，用来先拿频道集合，再做后续同步或联动处理。

## 5. 未读管理

```text
POST /conversations/clearUnread
POST /conversations/setUnread
```

两者分别适合：

- 用户读完会话后清空未读
- 业务或客户端特殊场景下主动设置未读数

## 6. 删除会话

```text
POST /conversations/delete
```

它管理的是“会话视图上的删除状态”，不是把消息物理删除掉。

## 这组接口最容易混淆的边界

### 会话同步 vs 消息同步

- 会话同步：拿最近会话列表和摘要
- 消息同步：按频道拿历史消息

所以：

- 进会话列表：先看 `/conversation/sync`
- 点进某个聊天页补历史：再看 `/channel/messagesync`

## 常见坑

### 1. 用消息同步代替会话同步

这样客户端会话列表的未读和摘要很容易自己算乱。

### 2. 把删除会话理解成删除消息

会话删除更多是用户视图层动作，不等于消息存储被清掉。

### 3. 单聊频道没有正确处理真实频道 ID

当前实现里，单聊内部会有真实频道 ID 转换逻辑，不能简单把所有单聊都按表面 `channel_id` 处理。

## 建议的接入顺序

1. 先接 `/conversation/sync`
2. 再补 `/conversations/clearUnread`
3. 需要删除能力时再接 `/conversations/delete`
4. 复杂同步场景再看 `syncByChannels` 和 `syncMessages`

核心接口：

```text
POST /channel/messagesync
```

这个接口用来按消息序号范围拉取频道历史消息，通常用于客户端补拉、历史同步或断线恢复。

你可以把它理解成“按游标翻页拉消息”的接口，而不是“按消息 ID 精确查询”的接口。

## 最重要的参数

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `login_uid` | 是 | 当前登录用户 UID |
| `channel_id` | 是 | 频道 ID |
| `channel_type` | 是 | 频道类型 |
| `start_message_seq` | 否 | 起始消息序号，包含 |
| `end_message_seq` | 否 | 结束消息序号，不包含 |
| `limit` | 否 | 返回条数上限 |
| `pull_mode` | 否 | 拉取方向，`0` 向下拉取，`1` 向上拉取 |
| `event_summary_mode` | 否 | 事件摘要模式，可选 `basic` / `full` |

其中最关键的是三件事：

- 你要拉哪个频道
- 你从哪个 `message_seq` 开始拉
- 你是往前翻还是往后翻

## 最小示例

```bash
curl -X POST "http://localhost:5001/channel/messagesync" \
  -H "Content-Type: application/json" \
  -d '{
    "login_uid": "user123",
    "channel_id": "group123",
    "channel_type": 2,
    "start_message_seq": 1000,
    "end_message_seq": 1100,
    "limit": 50,
    "pull_mode": 0
  }'
```

## 返回什么

返回结果不是单纯的消息数组，而是一个带翻页状态的对象：

```json
{
  "start_message_seq": 1000,
  "end_message_seq": 1100,
  "more": 1,
  "messages": []
}
```

其中：

- `messages` 是本次返回的消息列表
- `more` 表示是否还有更多数据
- `start_message_seq` / `end_message_seq` 是本次查询边界，便于客户端继续翻页

每条消息里通常会包含这类字段：

- `message_id`
- `message_seq`
- `from_uid`
- `timestamp`
- `payload`

## 最常见的 3 种用法

### 1. 打开会话时先拉最近一页

如果 `start_message_seq` 和 `end_message_seq` 都是 `0`，服务端会返回该频道最近的一批消息。  
这是最常见的“首次打开会话”用法。

### 2. 上滑继续加载更早的历史消息

已知本地最早一条消息序号后，可以继续按序号范围往前翻页。  
核心思路是：把当前最早的 `message_seq` 当作边界，继续拉更旧的数据。

### 3. 断线重连后补拉缺失消息

客户端记录本地最后一条已处理消息序号，重连后按这个游标向后补拉，就能把断线期间缺失的消息补回来。

## 使用建议

- 不要一次拉太多，优先分页
- `limit` 要控制住，避免一次返回过大；服务端也会限制最大值
- 用 `message_seq` 做客户端本地游标
- 把它和客户端会话、离线补拉逻辑配合起来用

## 它和其他消息接口怎么区分

- 拉历史、分页、断线恢复：用当前这个 `/channel/messagesync`
- 精确查某一条消息：看 [`查询单条消息`](/server/message/search)
- 在频道里写入一条新消息：看 [`发送消息`](/server/message/send)

不要把它当成“按消息 ID 搜索”的接口，它的设计重点就是按 `message_seq` 做范围同步。

## 什么时候会用它

- 用户重连后补消息
- 打开会话时拉历史记录
- 服务端或管理侧按范围排查某段消息

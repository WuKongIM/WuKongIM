WuKongIM 里最容易混淆的不是参数，而是“到底该查哪一种消息”。

这一组能力建议这样理解：

- **精确查一条**：`POST /message`
- **精确查多条**：`POST /messages`
- **按序号范围拉历史**：`POST /channel/messagesync`

如果你只是想拿到某几条确定的消息，优先看前两个接口；如果你是在做会话补拉、断线恢复、翻页历史，应该看 [`消息同步`](/server/message/sync)。

## 1. 查询单条消息

核心接口：

```text
POST /message
```

最常见用法是：

- 已知 `message_id`
- 或者已知 `client_msg_no`
- 想拿到该消息完整内容

### 最重要的参数

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `channel_id` | 是 | 频道 ID |
| `channel_type` | 是 | 频道类型 |
| `message_id` / `client_msg_no` | 二选一 | 精确定位消息 |
| `login_uid` | 个人频道时必填 | 用于定位单聊对应的真实频道 |

### 最小示例

```bash
curl -X POST "http://localhost:5001/message" \
  -H "Content-Type: application/json" \
  -d '{
    "login_uid": "u1001",
    "channel_id": "u1002",
    "channel_type": 1,
    "message_id": 123456789
  }'
```

### 返回什么

成功时返回单条消息对象，常见字段包括：

- `message_id`
- `message_seq`
- `client_msg_no`
- `from_uid`
- `channel_id`
- `channel_type`
- `payload`

## 2. 批量查询多条消息

核心接口：

```text
POST /messages
```

这个接口适合“你已经知道一批定位条件”，例如：

- 一组 `message_ids`
- 一组 `message_seqs`
- 一组 `client_msg_nos`

### 最重要的参数

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `channel_id` | 是 | 频道 ID |
| `channel_type` | 是 | 频道类型 |
| `login_uid` | 个人频道建议传 | 否则单聊查询容易定位错频道 |
| `message_ids` | 否 | 按消息 ID 查询 |
| `message_seqs` | 否 | 按消息序号查询 |
| `client_msg_nos` | 否 | 按客户端消息号查询 |

三组选项不要求只能传一种，但一般建议同一次请求只用一种定位方式，排查起来更清楚。

### 最小示例

```bash
curl -X POST "http://localhost:5001/messages" \
  -H "Content-Type: application/json" \
  -d '{
    "login_uid": "u1001",
    "channel_id": "u1002",
    "channel_type": 1,
    "message_ids": [123456789, 123456790]
  }'
```

### 返回结构

返回值不是裸数组，而是带 `messages` 字段的对象：

```json
{
  "start_message_seq": 0,
  "end_message_seq": 0,
  "more": 0,
  "messages": [
    {
      "message_id": 123456789,
      "message_seq": 1001
    }
  ]
}
```

当前查询接口里，`start_message_seq`、`end_message_seq`、`more` 往往只是兼容同步响应结构；你真正关心的是 `messages`。

## 3. 不要把它和消息同步混用

`POST /channel/messagesync` 更适合：

- 会话首屏拉历史
- 断线重连补消息
- 按序号区间翻页

而 `/message`、`/messages` 更适合：

- 运营后台按消息 ID 排查
- 根据 `client_msg_no` 回查消息是否写入
- 精准读取某几条消息明细

## 常见坑

### 1. 个人频道查询漏传 `login_uid`

单聊在服务端内部会转换成真实频道 ID。

所以个人频道场景下，`login_uid` 不是可有可无，最好始终带上。

### 2. 只传 `message_id`，却没传频道信息

当前实现不是全局无条件搜索，仍然要求你给出频道上下文。

至少要带：

- `channel_id`
- `channel_type`

### 3. 把“查多条精确消息”写成“按历史范围拉取”

如果你已经知道目标消息 ID，直接用 `/messages`，不要再走 `/channel/messagesync`。

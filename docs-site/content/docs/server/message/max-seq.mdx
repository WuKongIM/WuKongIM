---
title: 最大消息序号
description: 判断某个频道目前同步到了哪里时，先看这个接口。
---

核心接口：

```text
GET /channel/max_message_seq
```

这个接口的用途很明确：

- 你想知道某个频道当前最新消息序号是多少
- 你要做消息同步前的边界判断
- 你要排查“客户端到底落后了多少条消息”

## 什么时候用它

- 客户端进入会话前，先判断有没有新消息
- 做历史消息同步游标校验
- 服务端排查某个频道的消息增长情况

它比直接拉历史消息更轻，因为这里只返回一个序号。

## 查询参数

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `channel_id` | 是 | 频道 ID |
| `channel_type` | 是 | 频道类型 |
| `login_uid` | 个人频道时必填 | 用于定位单聊真实频道 |

## 最小示例

```bash
curl "http://localhost:5001/channel/max_message_seq?channel_id=group123&channel_type=2"
```

如果是单聊：

```bash
curl "http://localhost:5001/channel/max_message_seq?channel_id=u1002&channel_type=1&login_uid=u1001"
```

## 返回什么

```json
{
  "message_seq": 1500
}
```

如果频道还没有消息，或者当前读取不到频道配置，常见结果会是：

```json
{
  "message_seq": 0
}
```

## 这个序号怎么理解

`message_seq` 可以直接理解成：

- 当前频道最后一条消息的序号
- 后续同步时的上边界参考值

如果你的客户端本地最后一条是 `1490`，服务端返回 `1500`，那就说明大概率还有 10 条需要补。

## 常见坑

### 1. 单聊场景漏传 `login_uid`

当前实现里，个人频道会转换成内部真实频道 ID。  
所以单聊时最好始终带上 `login_uid`。

### 2. 把它当成“最后一条消息详情接口”

这个接口只返回最大序号，不返回消息内容。

如果你要最后一条消息内容，应该看：

- `GET /channel/last_message`

如果你要拉一段历史消息，应该看：

- `POST /channel/messagesync`

### 3. 看到 `0` 就以为系统异常

`0` 也可能只是：

- 频道还没有消息
- 频道当前还没建立对应配置

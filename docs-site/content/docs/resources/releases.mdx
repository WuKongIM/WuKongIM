这一页不重复维护完整变更日志，只负责回答升级时最实际的三个问题：

- 去哪里看官方发布记录
- 升级前先看什么
- 升级后怎么验证

## 官方发布入口

WuKongIM 的版本发布记录以 GitHub Releases 为主。

- 仓库：`WuKongIM/WuKongIM`
- 入口：`https://github.com/WuKongIM/WuKongIM/releases`

## 什么时候你应该先来看这页

- 准备从一个稳定版本升级到新版本
- 想确认新版本有没有破坏性变更
- 想知道升级后最少要回归哪些链路

如果你还没真正跑起来当前版本，不要先盯发布记录，先回 [`快速开始`](/quick-start)。

## 升级前先看哪几件事

### 1. 看破坏性变更

重点关注：

- 配置项是否有变化
- API 或返回字段是否有变化
- 集群、迁移、Webhook、SDK 是否有兼容性说明

### 2. 看部署方式受不受影响

不同部署路径，升级重点不同：

- Docker：镜像版本和容器重启方式
- Linux：二进制替换、配置兼容、手动回滚
- Kubernetes：Chart 版本、values 变更、滚动升级策略

### 3. 看你自己的接入面

如果你的系统已经接了：

- 客户端 SDK
- 服务端 API
- Webhook
- 管理后台

那升级评估不能只看服务端本身，还要看这些接入面是否会跟着变。

## 最小升级检查表

如果你不想漏项，最少按这个顺序看：

1. 看 Releases 里的变更说明
2. 对照你的部署方式看升级页
3. 对照你的接入面看是否有 API、Webhook、SDK 影响
4. 先在测试环境验证，再进生产
5. 保留可回滚版本和配置

## 升级后至少验证什么

最少做这几项：

1. 节点启动正常
2. `health` 正常
3. 客户端能连上
4. 单发消息正常
5. 历史消息和会话同步正常

## 什么时候先不要急着升

- 还没看清破坏性变更
- 当前版本运行稳定，但没有足够回滚预案
- 生产环境没有先做预演验证

## 不同角色最该看什么

- 运维：重点看部署方式、回滚预案、`health` 和集群状态
- 后端：重点看 API、Webhook、迁移相关变更
- 客户端：重点看 SDK、鉴权链路、消息和会话同步是否受影响

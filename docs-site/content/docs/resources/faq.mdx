这页不追求“大而全”，只收最常见、最容易把新手卡住的问题。

## 1. 第一次上手应该先看哪条路径？

默认建议：

1. 先看 [`快速开始`](/quick-start)
2. 跑通 [`Docker 单机部署`](/quick-start/docker-single-node)
3. 再去看 [`客户端接入`](/client/sdk-selection) 或 [`服务端集成`](/server)

如果一开始就直接翻 API 参考，大概率会被配置、地址、鉴权这些上下文问题卡住。

## 2. `5001` 是客户端连接地址吗？

不是。

`5001` 是 HTTP API 端口。客户端真正连的是：

- TCP：通常 `5100`
- WebSocket：通常 `5200`

如果你不想在客户端写死地址，应该先看 [`获取地址`](/server/route/get-address)。

## 3. 为什么我能打开后台，但客户端连不上？

最常见的原因有三个：

- 只开了 API 端口，没开 TCP 或 WebSocket
- `external` 地址配置不对
- 网关/Nginx 转发只配了 HTTP，没配 WS 或 TCP

先排查：

- `/health` 是否正常
- `ws://...` 地址是否真能访问
- `route` 接口返回的地址是否正确

## 4. 服务端接口是不是默认完全不用鉴权？

不要这样理解。

当前仓库配置里已经能看到：

- `managerToken`
- `auth.superToken`
- `auth.kind`
- `jwt.secret`

所以更稳妥的理解是：

你需要按自己的接入方式明确好服务端认证策略，而不是假设“天然全开放”。

## 5. 为什么消息发送成功了，却查不到？

先看是不是查错了维度：

- 精确查某条消息：看 [`消息查询`](/server/message/search)
- 按历史范围补拉：看 [`消息同步`](/server/message/sync)

另外还要确认：

- 查询时的 `channel_id`、`channel_type` 是否正确
- 单聊场景有没有带 `login_uid`
- `client_msg_no` 是否真的和发送时一致

## 6. `/message/sendbatch` 是不是一次提交多条不同消息？

不是。

当前实现里它表示：

**同一条消息发给多个用户。**

如果你要发多条不同消息，请多次调用 [`发送消息`](/server/message/send)。

## 7. 单机和集群应该怎么选？

经验上可以直接这么判断：

- 先学习、先验证、先小规模上线：单机
- 追求高可用、容灾、后续持续扩容：集群

看完整对比可以直接去 [`单机还是集群`](/operations/single-node-vs-cluster)。

## 8. 扩容时为什么还要改 Nginx？

因为当前仓库里的 Docker 集群示例是通过 Nginx 统一转发：

- API
- TCP
- WebSocket
- Manager
- Demo

新增节点后，如果 upstream 不更新，新节点虽然可能已经加入集群，但入口流量不会分过去。

## 9. 文档里这么多页面，我最容易先踩的坑是什么？

通常就四个：

- 把 API 端口当成长连接端口
- 混淆监听地址和对外地址
- 没搞清消息“查询”和“同步”的区别
- 先看细节接口，没先跑通最小链路

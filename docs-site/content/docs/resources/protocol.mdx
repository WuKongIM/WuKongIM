协议这页不把完整低层细节塞进主阅读路径里，而是先帮你分清：

- 什么时候需要读协议
- 该看哪一份协议
- 如果只是正常接入，是不是根本不需要先读协议

## 先说结论

大多数接入方一开始**不需要先看协议细节**。

更常见的顺序是：

1. 先跑通部署
2. 先接 SDK 或服务端接口
3. 只有在自研客户端、协议网关、深度排错时，才回头看协议

## 当前公开的两类协议说明

目前最重要的协议入口有两个：

- 二进制协议：`https://githubim.com/server/advance/proto.html`
- JSON 协议（仅 WebSocket）：`https://githubim.com/server/advance/jsonrpc.html`

## 什么时候看二进制协议

适合这些场景：

- 自己实现客户端
- 需要理解底层收发包结构
- 需要排查连接建立、ACK、心跳、消息投递等低层问题

你通常会在这里看到这类内容：

- 固定头
- 可变头
- 控制报文类型
- 标志位定义
- 长度编码

## 什么时候看 JSON 协议

如果你更关注：

- WebSocket 场景
- 更容易调试和抓包的协议形式
- 快速理解消息结构

那 JSON 协议通常更容易上手。

## 谁最需要看协议

- 自研客户端团队
- 协议代理 / 网关开发者
- 性能与网络排障人员

如果你只是：

- 用官方 SDK
- 调服务端 HTTP API

那更应该先看接入文档而不是协议全文。

## 最容易判断错的地方

- 用官方 SDK，也一上来先啃协议全文
- 只是调 HTTP API，却把时间花在低层包结构上
- 遇到连接问题时，不先排地址、Token、部署和网络，直接怀疑协议本身

真正的经验通常是：

- 先排部署和连接
- 再排接入方式
- 最后才回到协议细节

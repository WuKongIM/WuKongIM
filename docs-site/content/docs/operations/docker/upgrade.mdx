---
title: 升级
description: 先备份数据目录和配置，再升级镜像版本并做最小回归验证。
---

Docker 升级最重要的原则只有一个：

**先保留回滚条件，再换镜像。**

对大多数团队来说，升级动作本身并不复杂，真正容易出问题的是：

- 升级前没备份
- 升级后没验证对外地址和核心链路
- 集群里节点版本和网关配置没有统一检查

## 升级前先做什么

至少先保存三样东西：

- 数据目录
- 当前 `docker-compose.yaml`
- 当前 `nginx.conf` 或其他网关配置

如果你跑的是集群，还建议记录当前节点列表和镜像版本。

## 单机升级怎么做

如果你使用的是根目录的 `docker-compose.yaml`，最直接的升级方式就是修改镜像标签，例如：

```yaml
services:
  wukongim:
    image: registry.cn-shanghai.aliyuncs.com/wukongim/wukongim:v2
```

然后执行：

```bash
docker compose pull
docker compose up -d
```

## 集群升级怎么做

如果你跑的是 `docker/cluster` 示例，升级思路仍然一样：

1. 统一调整各节点镜像版本
2. 拉取新镜像
3. 重新拉起服务
4. 验证网关、管理后台和客户端连接链路

如果你的生产环境对可用性要求高，建议按自己的运维策略分批滚动，而不是所有节点一次性同时切换。

## 升级后先验证什么

不要一上来就只看容器是不是 `Up`，先看真正的业务入口：

```bash
curl http://127.0.0.1:5001/health
```

然后再检查：

- 管理后台是否可打开
- Demo 是否还能登录
- WebSocket 是否还能连通
- 常用服务端接口是否返回正常

## 推荐的最小回归清单

- `/health` 正常
- `/route` 返回地址正常
- `/message/send` 可以成功写入一条消息
- 客户端能正常连接并收发

## 常见坑

### 1. 只改镜像，不记当前版本

这样升级失败时很难快速回滚。

### 2. 升级后忘了检查外部入口

尤其是这几个：

- `external` 地址配置
- Nginx upstream
- Demo 和 Manager 入口

### 3. 集群节点版本改得不一致

如果你是多节点部署，最好把版本收敛清楚，不要长期混着跑。

### 4. 把升级理解成“镜像拉下来就结束”

真正的完成标准是：

- 服务起来了
- 入口可访问
- 核心消息链路能跑通

---
title: 单机还是集群
description: 说明什么时候适合单机部署，什么时候应该直接上集群。
---

这一步真正要解决的不是“技术上能不能做集群”，而是：

**你现在有没有必要一开始就上集群。**

## 先给判断标准

### 先上单机，如果你符合这些条件

- 正在学习或做 Demo
- 业务规模还小
- 先想验证功能和接入链路
- 可以接受手动备份和暂时没有容灾

### 直接上集群，如果你符合这些条件

- 业务已经进入生产，不能接受单点故障
- 对数据安全和持续可用性要求高
- 预计在线规模会持续增长
- 团队已经有多台机器和基本运维能力

## 一句结论版

- 目标是学习、验证、快速上线：先单机
- 目标是高可用、容灾、持续扩容：直接集群

## 单机的优缺点

优点：

- 部署最简单
- 成本最低
- 性能通常已经足够早期业务使用
- 后续可以继续扩成集群

缺点：

- 没有容灾
- 需要自己关注备份和故障恢复

## 集群的优缺点

优点：

- 高可用
- 容灾能力更强
- 支持在线扩容
- 更适合正式生产

缺点：

- 需要多台机器
- 部署和运维复杂度更高

## 可以怎么选

| 当前阶段 | 建议 |
| --- | --- |
| 第一次试用 | 单机 |
| 小型项目刚开始上线 | 单机起步 |
| 已有稳定流量且重视容灾 | 集群 |
| 大规模生产或核心业务 | 集群 |

## 最常见的误判

- 业务还没跑起来，就先把复杂度花在集群上
- 明明已经是核心生产业务，却还把单机当长期方案
- 把“以后可能会扩”误解成“现在必须先上集群”

更稳妥的做法通常是：

- 先按当前阶段做最合适的部署
- 预留好后续演进路径
- 在真正出现高可用需求时再切到集群

## 推荐路径

1. 第一次接入先走单机
2. 把客户端接入、服务端集成、运维链路跑通
3. 再进入集群部署和扩容文档

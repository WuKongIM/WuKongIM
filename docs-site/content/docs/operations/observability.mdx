这页的目标很直接：

**把运维里最容易分散的“看状态、看指标、看异常”收拢成一条最小观测链路。**

如果你已经把 WuKongIM 跑起来了，但还不知道应该先盯哪些信号，就从这里开始。

## 先分清三类观测

### 1. 存活与健康

回答的是：

- 服务是不是活着
- 节点是不是异常
- 集群是不是可用

最直接的入口通常是：

- `health`
- 管理后台节点状态

### 2. 连接与流量

回答的是：

- 当前连接数怎么样
- 消息流量有没有异常波动
- 节点负载是否失衡

这是最能直接反映 IM 系统运行状态的一组数据。

### 3. 追踪与排障

回答的是：

- 某个监控图为什么突然异常
- 某个节点为什么数据不对
- 扩容或升级后到底是哪一层出问题

这类问题通常要结合 Prometheus、管理接口和运行日志一起看。

## 当前仓库里最值得先看的观测入口

从现有配置和文档看，先优先理解这几块：

- `trace.prometheusApiUrl`
- Prometheus 抓取配置
- 管理端口和管理后台
- 健康检查接口

也就是说，WuKongIM 的观测通常是这三层一起工作：

- Prometheus 负责抓
- WuKongIM 负责暴露和查询
- 管理面板负责展示

## 最推荐先盯住的几项

如果你刚开始做运维，不要一上来就追求大而全图表，先盯下面这些最有价值：

- 节点是否存活
- 当前连接数
- 消息发送和接收趋势
- 集群节点状态
- 扩容或升级后是否出现异常波动

## 最小观测链路怎么搭

最稳妥的顺序是：

1. 先让 `health` 和管理后台可访问
2. 再让 Prometheus 能抓到所有节点
3. 再确认 `trace.prometheusApiUrl` 指向正确
4. 最后再看图表、告警和长期留存

如果前两步都没打通，后面的“图表漂亮不漂亮”意义不大。

## 最常见的误区

- 只盯 CPU / 内存，不盯连接和消息链路
- 只看单节点，不看集群状态
- 只看容器活着，不看客户端是否真的能连

## 一个实用原则

运维里最常见的错误不是“没有监控”，而是：

- 只看 CPU / 内存，不看连接和消息链路
- 只看单节点，不看集群整体
- 只看容器存活，不看客户端是否真的可连

所以观测一定要围绕“客户端能不能连、消息能不能走、节点是不是稳定”来组织。

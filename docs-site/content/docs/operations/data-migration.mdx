这页只回答三个实际问题：

- 什么时候你真的会遇到迁移
- 迁移过程中先看哪里
- 怎么判断迁移已经结束，还是其实还在重试

## 什么时候需要关心迁移

不是每次部署都会遇到迁移。

你通常在这些场景里才需要重点看它：

- 老版本升级到需要迁移数据结构的新版本
- 启动后怀疑系统一直卡在迁移阶段
- 管理侧看到节点起来了，但业务链路仍然异常

## 迁移不是“接口触发”，而是启动过程里的任务

当前仓库实现里，迁移任务会在服务启动流程里执行，并按阶段推进。

从代码上看，主要阶段包括：

- `message`
- `user`
- `channel`

也就是说，你在运维里最常见的不是“手工点一下开始迁移”，而是：

- 服务启动
- 迁移任务开始跑
- 你通过结果接口和日志观察它是否顺利完成

## 第一观察入口：迁移结果接口

最先看的就是：

```text
GET /migrate/result
```

对应文档：[`数据迁移接口`](/server/system/migrate)

这个接口不会发起迁移，只负责告诉你当前迁移状态。

## 怎么理解迁移结果

当前实现里，最重要的几个字段是：

- `status`
- `step`
- `last_err`
- `try_count`

可以先这样理解：

- `status=running`
  还在迁移
- `status=completed`
  当前任务已停止，但不代表一定已经标记为最终迁移完成
- `status=migrated`
  数据目录已经被标记为迁移完成

如果结果长期停在某个 `step`，或者 `try_count` 一直增加，就应该立刻去对照日志看原因。

## 第二观察入口：服务日志

迁移结果接口只能告诉你“大概卡在哪”，真正的细节还是在日志里。

最实用的排查方式是：

1. 先看 `/migrate/result`
2. 确认当前卡在 `message`、`user` 还是 `channel`
3. 再到服务日志里看对应错误

如果你跳过这一步直接重启，很多时候只会把问题重复一遍。

## 怎么判断迁移已经真的结束

不要只看服务是否已经启动，还要同时确认：

1. `/migrate/result` 不再处于 `running`
2. `/health` 正常
3. 实际消息、用户、频道相关链路可用

因为“进程起来了”和“迁移已经完成”不是同一个判断。

## 迁移期间最稳妥的做法

- 不要频繁反复重启
- 先保留日志
- 先看 `last_err`
- 明确当前是否仍在重试

尤其是当 `try_count` 持续增加时，说明系统还在循环尝试当前阶段。

## 最常见的误区

### 误区 1：迁移页就是升级页

不是。升级是更大的运维动作，迁移只是升级过程中可能出现的一步。

### 误区 2：结果接口会触发迁移

不会，它只是查看结果。

### 误区 3：`migrated` 之前就可以默认一切正常

不行。最好仍然做一次最小业务链路验证。

## 推荐的排查顺序

1. 看 `/migrate/result`
2. 看服务日志
3. 看 `/health`
4. 看客户端连接
5. 看消息与会话链路

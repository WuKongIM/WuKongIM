这页不按“错误码大全”写，而是按实际最常见的故障路径写。

先记一个原则：

**WuKongIM 出问题时，先判断是部署层、连接层、消息层，还是集群层。不要一上来就全查。**

## 第一步：先判断问题落在哪一层

### 现象 A：服务根本起不来

优先查：

- 配置文件
- 端口占用
- 数据目录权限
- 启动日志

### 现象 B：服务起来了，但客户端连不上

优先查：

- `health`
- 路由接口
- 对外地址配置
- WebSocket / TCP 入口
- `uid` / `token`

### 现象 C：能连接，但消息不正常

优先查：

- 频道参数
- 发送状态监听
- 消息同步链路
- Webhook 或业务服务端逻辑

### 现象 D：单机正常，集群异常

优先查：

- 节点互通
- `serverAddr`
- `apiUrl`
- `initNodes` 或 `seed`
- 统一入口层

## 第二步：按最小顺序排

很多问题只要按这个顺序查，就能少走很多弯路：

1. 先看进程或容器是否真的活着
2. 再看 `health`
3. 再看管理后台或节点状态
4. 再看客户端连接
5. 最后才看消息和会话细节

## 最常见的几类问题

## 1. 健康检查不正常

通常说明问题还停留在服务层，不要先去查客户端 SDK。

优先看：

- 启动日志
- 配置项
- 端口监听
- 数据目录

## 2. 路由返回地址不对

最常见原因是：

- 对外地址配置不对
- 环境变量覆盖了配置文件
- 网关或统一入口地址没更新

这类问题经常表现为：

- API 能访问
- 客户端还是连不上

## 3. WebSocket 能连，但消息不同步

这时问题大概率已经不在“连接是否成功”，而在：

- 频道上下文
- 消息同步接口
- 会话同步回调
- 业务服务端数据源

如果你用的是完整 SDK，这一步尤其常见。

## 4. 集群扩容后状态不对

优先查：

- 新节点 `nodeId` 是否唯一
- `serverAddr` / `apiUrl` 是否内网可达
- 网关有没有把新节点加进去
- Prometheus 抓取目标有没有补

## 5. 升级后客户端异常

不要只看节点是不是启动成功，还要实际验证：

- `/route`
- WebSocket 连接
- 单条消息发送
- 会话和历史消息同步

## 一份最实用的排查清单

如果你现在没头绪，直接按这个顺序查：

1. `health` 正不正常
2. 管理后台能不能打开
3. 路由接口返回什么地址
4. 客户端实际连的是哪个地址
5. 发送一条消息是否成功
6. 会话和历史消息是否正常

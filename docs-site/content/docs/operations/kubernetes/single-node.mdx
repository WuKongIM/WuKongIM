---
title: 单机部署
description: 基于 Helm 在 Kubernetes 中部署单节点 WuKongIM。
---

如果你的团队已经有 Kubernetes 基础设施，但当前目标只是：

- 先跑通环境
- 验证接入链路
- 在测试环境里先落一个最小实例

那么 Kubernetes 单机部署是这条路径里最轻的一种方式。

## 先给判断

如果你只是第一次试用 WuKongIM，仍然更推荐先看 Docker 单机。

Kubernetes 单机更适合这类团队：

- 已经在 K8s 上托管其他业务
- 希望部署方式和现有平台统一
- 后续大概率会扩到多副本或集群

## 这条路径的前提

- 已有可用 Kubernetes 集群
- 已配置 `kubectl`
- 已安装 Helm 3
- 有可用持久化存储能力

## 当前仓库里的现实情况

当前主仓库没有直接内置本地 Helm Chart 或 Kubernetes 清单，这条部署路径延续的是官方 Helm 仓库方式。

所以真正落地时，建议先确认：

- Helm 仓库可访问
- Chart 版本与你要部署的服务版本匹配
- 集群存储和 Service 暴露方式符合你的环境

## 基本步骤

### 1. 添加 Helm 仓库

```bash
helm repo add wukongim https://wukongim.github.io/helm/
helm repo update
helm search repo wukongim
```

### 2. 选择 Chart 版本后安装

```bash
helm install wkim wukongim/wukongim \
  -n wukongim \
  --create-namespace \
  --set replicaCount=1
```

如果你需要固定 Chart 版本，再额外加上：

```bash
--version <chart-version>
```

## 部署后先做什么检查

```bash
kubectl get pods -n wukongim
kubectl get svc -n wukongim
kubectl get pvc -n wukongim
helm status wkim -n wukongim
```

最重要的是确认三件事：

- Pod 正常 Running
- 持久化卷已绑定
- Service 已正确暴露 API / WebSocket 所需端口

## 怎么验证服务可用

如果只是本地验证，最简单的是先做端口转发：

```bash
kubectl port-forward svc/<service-name> 5001:5001 -n wukongim
kubectl port-forward svc/<service-name> 5200:5200 -n wukongim
```

然后再验证：

- `http://127.0.0.1:5001/health`
- `ws://127.0.0.1:5200`

这里的 `<service-name>` 取决于你的 release 名称和 chart 生成结果，先用 `kubectl get svc -n wukongim` 确认。

## 最常见的坑

- 只会部署 Pod，不检查持久化卷是否真的绑定
- 直接把生产暴露方式照搬到本地验证环境
- 忽略外部地址、Service 类型、Ingress 或负载均衡配置
- 以为 K8s 单机就等于高可用

title: 集群配置
description: 先把 nodeId、节点通信地址、apiUrl、initNodes 和 seed 的角色分清。

集群配置难的不是参数数量，而是这些参数都长得很像，但含义完全不同。

如果你先把下面五个东西理顺，集群配置就已经过了最难的一关：

- `nodeId`
- `addr`
- `serverAddr`
- `apiUrl`
- `initNodes` / `seed`

## 1. 先看现实样例

当前仓库最值得先对照的是：

- `exampleconfig/cluster1.yaml`
- `exampleconfig/cluster2.yaml`
- `exampleconfig/cluster3.yaml`
- `exampleconfig/join_as_replica.yaml`
- `exampleconfig/join_as_proxy.yaml`

前 3 份更适合理解“初始集群怎么搭”，后 2 份更适合理解“后续节点怎么加入”。

## 2. 最关键的配置块

可以先按这个心智模型读：

```yaml
cluster:
  nodeId: 1
  addr: "tcp://127.0.0.1:10001"
  serverAddr: "127.0.0.1:10001"
  apiUrl: "http://127.0.0.1:5001"
  initNodes:
    - "1@127.0.0.1:10001"
    - "2@127.0.0.1:10002"
    - "3@127.0.0.1:10003"
```

## 3. 这些字段怎么理解

### `nodeId`

当前节点唯一编号，集群里不能重复。

### `addr`

当前节点真正监听的集群通信地址。

### `serverAddr`

其他节点访问当前节点时使用的地址。它必须是其他节点可达的内网地址。

### `apiUrl`

其他节点访问当前节点 HTTP API 时使用的地址，也应该是节点间可达地址。

### `initNodes`

初始集群节点清单。用于第一次 bootstrap 集群时的全量初始节点配置。

## 4. `initNodes` 和 `seed` 最容易混淆

可以简单这样记：

- `initNodes`：更适合初始集群启动
- `seed`：更适合已有集群后的新节点加入

也就是说：

- 新建固定 3 节点集群，优先看 `initNodes`
- 后面再加新节点，优先看 `seed`

## 5. 节点间地址和客户端入口不是一回事

这是集群里最容易配错的核心点。

- `serverAddr` / `apiUrl`：给集群内部用
- `external.*` 或统一入口地址：给客户端和业务系统用

很多“节点都活着，但客户端还是不通”的问题，本质上就是把这两层地址混掉了。

## 6. 推荐配置顺序

最稳妥的顺序是：

1. 给每个节点分配唯一 `nodeId`
2. 确认每个 `serverAddr` 在节点间可达
3. 确认每个 `apiUrl` 在节点间可达
4. 确认初始集群到底用 `initNodes` 还是后续加入用 `seed`
5. 最后再处理统一入口和对外地址

## 7. 最常见的错误

### 错误 1：`nodeId` 重复

这是最直接、最危险的一类错误。

### 错误 2：`serverAddr` 写成只有本机能访问的地址

例如写成错误回环地址或不可达地址，表面上节点能启动，实际上节点间不能协作。

### 错误 3：`apiUrl` 配成公网入口

它更应该先是节点间稳定可达的地址，而不是给外部系统用的统一入口。

### 错误 4：初始集群节点配置不一致

不同节点看到的 `initNodes` 不一致，后面会很难排查。

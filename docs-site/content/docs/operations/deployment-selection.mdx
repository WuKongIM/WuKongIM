这一页的目标很直接：先帮你确定该走哪条部署路径，而不是一上来就在 Docker、Linux、Kubernetes 之间反复跳。

## 先给推荐

### 第一次接触或小型项目

优先选 **Docker 单机部署**。

原因：

- 上手最快
- 部署步骤最少
- 验证成本最低
- 后续仍然可以扩成集群

### 已准备生产、对容灾有要求

优先选 **集群部署**。

原因：

- 有高可用和容灾能力
- 支持在线扩容
- 更适合正式业务持续演进

### 对性能特别敏感、能接受更高运维介入

考虑 **Linux 直接部署**。

原因：

- 资源占用更低
- 路径更直接
- 更适合对运行环境有强控制能力的团队

### 云原生团队或已有 Kubernetes 基础设施

考虑 **Kubernetes 部署**。

原因：

- 与现有平台治理方式更一致
- 更适合规模化运维
- 但理解和运维成本会更高

## 单机还是集群

| 你的情况 | 建议 |
| --- | --- |
| 学习、演示、早期验证 | 单机 |
| 中小业务，先上线再扩展 | 单机起步 |
| 对数据安全和容灾要求高 | 直接集群 |
| 预计在线规模持续增长 | 集群 |

最重要的判断是：**单机不是死路**。  
如果你现在更关心“先跑起来”，完全可以先走单机，再在明确需要高可用时切到集群。

## 方案对比

| 方案 | 部署难度 | 运维复杂度 | 性能 | 适用场景 |
| --- | --- | --- | --- | --- |
| Docker | 低 | 低 | 良好 | 新手、演示、中小型应用 |
| Linux | 低到中 | 中 | 最优 | 性能敏感、环境可控 |
| Kubernetes | 中到高 | 中到高 | 良好 | 云原生生产环境 |

## 最省时间的决策方法

不用一次比较所有维度，按下面 3 个问题判断就够了：

1. 你现在更关心“先上线”还是“先高可用”
2. 你的团队更熟 Docker、Linux 直跑，还是 Kubernetes
3. 你是否已经有固定的基础设施约束

## 推荐的决策顺序

1. 先决定是不是要高可用
2. 再决定团队更熟悉 Docker、Linux 还是 Kubernetes
3. 最后再进入对应方案的部署文档

## 常用端口

| 端口 | 用途 | 备注 |
| --- | --- | --- |
| `5001` | HTTP API | 建议仅内网开放 |
| `5100` | TCP 连接 | 客户端连接 |
| `5200` | WebSocket 连接 | Web 客户端连接 |
| `5300` | 管理后台 | 建议受控访问 |
| `5172` | Demo | 体验环境使用 |
| `11110` | 集群节点通信 | 节点内网通信 |

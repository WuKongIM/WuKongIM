---
title: 升级
description: Linux 二进制升级的核心是保留回滚条件，再替换可执行文件并做链路验证。
---

Linux 直跑升级比 Docker 少了一层镜像管理，但本质上并没有更“轻松”。

最容易出问题的地方是：

- 直接覆盖二进制，没有保留旧版本
- 配置文件和数据目录没备份
- 升级后只看进程起来了，没有看连接链路

## 最重要的原则

**先能回滚，再做替换。**

对 Linux 直跑来说，至少要保留：

- 当前可执行文件
- 当前配置文件
- 当前数据目录

## 升级前建议备份什么

- `wukongim` 可执行文件
- 你的 `wk.yaml` 或其他实际配置文件
- `rootDir` 对应的数据目录

如果是多节点部署，还建议记录每个节点当前的版本和角色。

## 单机升级怎么做

最直接的方式是用新版本二进制替换旧版本二进制。

例如：

```bash
mv wukongim wukongim.bak
cp ./new-version/wukongim ./wukongim
chmod +x ./wukongim
```

然后按你的运行方式重启服务。

如果你是手工后台运行：

```bash
./wukongim stop
./wukongim --config ./wk.yaml -d
```

## 集群升级怎么理解

如果你跑的是 Linux 多节点，不建议把升级理解成“所有节点一起替换二进制”。

更稳妥的做法是：

1. 先确认备份和回滚方案
2. 逐节点替换二进制
3. 每次替换后验证该节点与入口链路
4. 再继续下一个节点

这样一旦某一节点异常，问题范围更小。

## 升级后先验证什么

优先按这条顺序检查：

### 1. 健康检查

```bash
curl http://127.0.0.1:5001/health
```

### 2. 管理后台

```text
http://<host>:5300/web
```

### 3. Demo 或客户端连接

确认：

- WebSocket 能连
- TCP 能连
- 发送一条消息能成功

## 推荐最小回归清单

- `/health` 返回正常
- `/route` 返回的连接地址正确
- `/message/send` 发送成功
- 管理后台能登录
- 客户端能连接并收发

## 常见坑

### 1. 替换了二进制，但忘了 `chmod +x`

这在手工升级时非常常见。

### 2. 升级后只验证 API，不验证长连接

WuKongIM 的核心价值在长连接和消息链路，只看 HTTP 不够。

### 3. 多节点升级时没有逐步验证

如果所有节点一起替换，出问题时很难快速定位到底是哪一层出了偏差。

### 4. 配置和版本不配套

升级前最好重新对照一次当前仓库示例配置，尤其是鉴权、集群和对外地址相关参数。

title: 扩容
description: Linux 直跑二进制时，新增节点不只是加进程，还要同步更新接入地址和网关入口。
---

Linux 扩容最容易被误解成“把程序再跑一份就行了”，但真正要完成的是：

- 新节点加入集群
- 原有节点知道新拓扑
- 客户端入口能把流量转过去

所以这页不按“命令清单”写，而是按实际扩容路径写。

## 先分两类场景

### 场景 A：从单机扩成多节点

你原来只有一个单机节点，现在准备加第二台或更多节点。

### 场景 B：现有集群继续加节点

你已经有多节点集群，现在准备继续增加副本或扩容角色。

这两类的共同点是：都不能只在新机器上启动进程就结束。

## 当前仓库里可参考的配置入口

如果你做 Linux 直跑部署，当前仓库里最值得对照的是：

- `exampleconfig/single.yaml`
- `exampleconfig/cluster1.yaml`
- `exampleconfig/cluster2.yaml`
- `exampleconfig/cluster3.yaml`
- `exampleconfig/join_as_replica.yaml`
- `exampleconfig/join_as_proxy.yaml`

它们刚好覆盖了：

- 单机
- 初始多节点集群
- 新节点加入集群
- 代理节点加入集群

## 场景 A：单机扩成多节点

### 第一步：确认原节点不再继续暴露“单机直连地址”

一旦你准备让多节点对外服务，就要开始区分：

- 节点自身监听地址
- 节点间通信地址
- 对外统一入口地址

也就是说，客户端最好不再直连某个节点的私有地址，而是转到统一入口层。

### 第二步：为新节点准备独立配置

新节点至少要有独立的：

- `cluster.nodeId`
- `cluster.addr`
- `cluster.serverAddr`
- `cluster.apiUrl`
- `rootDir`

不能和原节点复用。

### 第三步：决定加入方式

当前仓库里有两种你会实际遇到的方式：

- 初始固定集群：使用 `initNodes`
- 后续新节点加入：使用 `seed`

如果你是在新建一套 3 节点集群，优先参考 `cluster1.yaml`、`cluster2.yaml`、`cluster3.yaml`。  
如果你是已经有集群，再新增节点，更贴近 `join_as_replica.yaml` 这种写法。

### 第四步：给入口层补上新节点

不管你用 Nginx 还是其他网关，至少要把这些入口同步加上新节点：

- API
- TCP
- WebSocket
- 管理后台
- Demo

否则新节点虽然可能已经在集群里，但实际流量不会过去。

## 场景 B：现有集群继续加节点

### 推荐先看哪份示例

如果你只是新增一个普通数据节点，优先参考：

```text
exampleconfig/join_as_replica.yaml
```

如果你准备新增代理节点，参考：

```text
exampleconfig/join_as_proxy.yaml
```

### `join_as_replica.yaml` 里最关键的字段

```yaml
cluster:
  nodeId: 5
  addr: "tcp://127.0.0.1:10005"
  serverAddr: "127.0.0.1:10005"
  apiUrl: "http://127.0.0.1:5005"
  seed: "1@127.0.0.1:10001"
```

怎么理解：

- `nodeId`：新节点唯一编号
- `serverAddr`：其他节点能访问它的内网地址
- `apiUrl`：其他节点能访问它的 HTTP 地址
- `seed`：已有集群中的任意一个已加入节点

这里的 `seed` 更适合描述“后续加入集群”的入口点。

## 操作顺序建议

最稳妥的顺序是：

1. 先备份旧节点数据和配置
2. 在新节点准备独立配置文件
3. 启动新节点进程
4. 更新网关 upstream
5. 更新监控采集目标
6. 再做管理后台和链路验证

## 启动方式

Linux 直跑的最小方式仍然是：

```bash
./wukongim --config ./your-node.yaml -d
```

如果是 systemd 或其他进程托管方式，就按你自己的运维体系执行，但本质上仍然是加载新的节点配置。

## 怎么验证扩容成功

至少看这几项：

- 新节点进程正常启动
- 健康检查正常
- 管理后台节点列表出现新节点
- 客户端通过统一入口还能正常连接
- 新节点已被监控系统采集到

## 常见坑

### 1. 复用了旧节点的 `nodeId`

这会直接把集群状态搞乱，`nodeId` 必须唯一。

### 2. 新节点本机能启动，但其他节点连不到

通常是 `serverAddr` 或 `apiUrl` 配成了本机可见、但其他机器不可达的地址。

### 3. 只改了新节点，没改网关

这样扩容只是“集群变大了”，不是“入口能力变大了”。

### 4. 混淆 `initNodes` 和 `seed`

- `initNodes` 更适合初始集群 bootstrap
- `seed` 更适合后续节点加入

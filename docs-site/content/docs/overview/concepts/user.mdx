用户是你业务系统里的身份主体，也是 WuKongIM 做鉴权、路由和多端同步时最基础的对象。

## 先把“用户”和另外两个概念分开

很多第一次接入的人，会把这三个词混在一起：

- 用户
- 设备
- 连接

它们的关系应该这样理解：

- 用户：业务身份，比如 `u10001`
- 设备：这个用户登录的某个终端，比如 iPhone、Android、Web
- 连接：某个设备当前和 WuKongIM 建立的一条长连接

**一个用户可以有多个设备，一个设备在某个时刻通常对应一条活跃连接。**

## 在 WuKongIM 里，用户通常承担什么角色

### 1. 鉴权主体

客户端连接前，通常要拿到这个用户的 `uid` 和 `token`。

### 2. 消息收发主体

无论是单聊还是系统通知，最终都绕不开“这个消息和哪个用户有关”。

### 3. 多端同步主体

一个用户可能同时在 Web、iOS、Android 登录，WuKongIM 需要围绕这个用户做消息投递和状态同步。

## 用户 ID 由谁决定

由你的业务系统决定，不是 WuKongIM 自动生成。

文档里出现的 `uid` 本质上是“你自己的用户标识”，所以接入时要保证：

- 全局唯一
- 长期稳定
- 不要在不同业务环境里混用

## token 的角色是什么

`token` 用来证明“这个连接确实代表这个用户”。

WuKongIM 本身不是你的用户中心，所以通常是：

1. 用户先登录你的业务系统
2. 你的服务端生成或下发连接凭证
3. 客户端拿着 `uid` 和 `token` 去连接 WuKongIM

这也是为什么新文档把“客户端接入”和“服务端鉴权与连接”拆开写。

## 最容易犯错的地方

### 误区 1：把用户当成设备

同一个用户多端登录是常态，所以用户层状态和设备层状态不能混写。

### 误区 2：把 WuKongIM 当成用户系统

它负责即时通信基础设施，不负责替你管理完整账号体系、业务资料和权限模型。

### 误区 3：连接地址可以固定死给客户端

生产环境里，用户连接到哪个节点，往往需要先通过路由接口获取。

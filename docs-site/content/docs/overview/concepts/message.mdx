消息是 WuKongIM 里真正被传输、存储、同步的核心对象。

如果只记一件事，可以记这句话：

**用户不是直接把消息发给“连接”，而是把消息发到某个频道。**

## 一条消息最重要的几部分

- 发到哪个频道：由 `channel_id` 和 `channel_type` 一起确定
- 谁发的：发送者用户标识
- 发了什么：消息内容 `payload`
- 怎么处理：消息头决定是否持久化、是否计入红点、是否只同步一次
- 在频道里的位置：消息会有自己的顺序编号，用来做同步和补拉

## 为什么消息一定要和频道绑定

WuKongIM 的很多能力，都是围绕“频道里的消息流”建立的：

- 消息投递
- 离线补拉
- 历史查询
- 会话未读计算
- 多端同步

所以你在后面的接口文档里会反复看到 `channel_id` 和 `channel_type`，这不是重复设计，而是消息定位的基本条件。

## 单聊消息其实也是发到频道

这一点很容易误解。

即使是一对一聊天，WuKongIM 也不是“直接给某个 TCP 连接发包”，而是发到一个个人频道。这样做的好处是：

- 多设备登录时可以统一处理同步
- 离线消息和历史消息模型一致
- 服务端接口和客户端 SDK 不需要区分两套完全不同的消息通路

## 你在业务里通常会关心什么

### 1. 发送

服务端主动发消息，或者客户端通过 SDK 发消息。

### 2. 同步

客户端首次登录、重连、切设备时，需要知道从哪里开始补拉。

### 3. 查询

后台管理、审计、客服或排障时，往往需要按频道上下文查消息。

### 4. 状态

有些消息需要持久化，有些是临时指令；有些需要红点，有些不需要。这些差异通常由消息头或业务约定决定。

## 最常见的三个误区

### 误区 1：消息就是一段文本

不是。文本只是消息内容的一种表现形式。图片、命令、流式事件、自定义业务消息，本质上也都是消息。

### 误区 2：只要知道消息 ID 就能完成所有查询

实际排障和接口调用里，消息通常仍然需要放回频道上下文里理解和定位。

### 误区 3：消息发送成功就等于业务完成

不一定。你还要看是否需要持久化、是否要进入会话、是否要触发 Webhook、是否需要给离线设备补拉。

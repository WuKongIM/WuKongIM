WuKongIM 是一个面向实时通信场景的分布式即时通讯基础设施，重点解决三类问题：

- 长连接和消息通道如何稳定承载大规模在线用户
- 消息如何可靠存储、同步和回放
- 系统如何在单机起步后继续扩容到集群

一句话概括它的设计取向：

**像 Redis 一样简单，像 Kafka 一样高性能，像 MySQL 一样可靠。**

## 它适合什么场景

WuKongIM 适合你把“即时通信能力”当成系统基础设施来建设，而不是只临时发几条通知。

常见场景包括：

- 聊天应用：单聊、群聊、社区、私信
- 消息推送：需要稳定在线连接和离线补偿的通知系统
- IoT 通信：设备到设备、设备到服务端的实时消息
- 音视频信令：WebRTC 或实时互动场景的信令层
- 直播互动：弹幕、聊天室、互动消息
- 客服系统：坐席、访客、会话流转
- AI 通信：Agent、机器人、消息流式协作

## 什么时候值得用它

如果你的系统已经明确需要下面这些能力，WuKongIM 会比“自己拼装长连接网关 + 存储 + 同步逻辑”更合适：

- 多端在线和消息同步
- 离线消息与历史消息查询
- 大群、海量连接或持续增长的在线规模
- 单机起步，后续扩成高可用集群
- 需要同时对接客户端 SDK 和服务端 API

## 什么时候先不要急着上

如果你当前只需要下面这些能力，WuKongIM 可能还不是第一优先级：

- 简单的业务通知，没有长连接要求
- 不需要消息持久化和会话同步
- 业务体量很小，只想做临时 Demo，不准备继续演进

这时你可以先跑一遍 [`快速开始`](/quick-start)，确认系统模型是否符合你的产品。

## 先理解的三个判断标准

### 1. 部署门槛

WuKongIM 可以从单机起步，不要求先搭第三方中间件。你可以先跑通最小链路，再决定是否走集群。

### 2. 业务边界

它偏向“消息基础设施”，不是 UI 组件库，也不是替你完成整个 IM 业务层。用户体系、业务鉴权和上层产品逻辑仍然由你的系统掌控。

### 3. 扩展路径

单机不是死路。只要前期按推荐方式配置，后续可以平滑进入集群化部署。

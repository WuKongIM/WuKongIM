WuKongIM 可以先按一个很简单的模型理解：

- **频道** 是通信的核心单元
- **用户** 通过客户端连接到节点
- **消息** 发布到频道，再分发给订阅者
- **节点** 在集群中协同工作，负责高可用和扩展

如果你第一次看架构页，先不要把它当成“实现细节页”，而要把它当成：

- 你后面读客户端文档时的地图
- 你后面读服务端接口时的地图
- 你后面读部署和集群文档时的地图

## 先看整体关系

```text
业务系统
  ├─ 负责用户体系、业务鉴权、业务逻辑
  └─ 通过服务端 API / Webhook 与 WuKongIM 配合

WuKongIM
  ├─ 负责连接、消息、频道、会话、同步
  ├─ 提供 TCP / WebSocket / HTTP API / 管理端
  └─ 可从单机演进为集群

客户端
  ├─ 通过 SDK 建立长连接
  └─ 发送、接收、同步消息与会话
```

## 你只要先抓住 4 条链路

### 客户端连接链路

客户端通过 TCP 或 WebSocket 连接 WuKongIM。对外地址通常由节点配置中的 `external` 或对外端口决定。

### 业务对接链路

你的业务后端不直接替代 WuKongIM，而是通过服务端 API 管理用户 token、消息、频道、会话等能力。

### 消息流转链路

消息进入系统后会围绕频道、订阅关系、在线状态和同步状态流转，这也是后续概念文档的核心。

### 运维链路

单机可以直接体验；进入生产后再考虑集群、监控、升级和迁移。

## 看完这页后最重要的理解

- WuKongIM 不是你的业务后端替身
- 客户端不是直接连“某个消息对象”，而是围绕频道和连接工作
- 服务端 API、Webhook、SDK、部署，其实都在服务同一套模型
